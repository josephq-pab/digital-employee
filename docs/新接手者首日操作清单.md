# 新接手者首日操作清单

> 版本：v1.0（2026-04-16）
> 适用：任意新 Agent 或新人员接手本项目
> 目标：第一天能运行、能验证、能知道边界

---

## 一、第一天读什么

### 第一天必读（按顺序）

| # | 文档 | 预计时间 | 读完应知道 |
|---|------|---------|---------|
| 1 | `RUNBOOK.md`（在根目录） | 15分钟 | 项目能做什么、怎么运行、输出在哪里 |
| 2 | `docs/稳定基线版本统一说明.md` | 5分钟 | 当前稳定基线是哪个版本 |
| 3 | `project/docs/新Agent开工清单.md` 第1-6节 | 20分钟 | 项目全貌和开工边界 |
| 4 | `project/docs/输入输出数据契约.md` | 15分钟 | 输入字段和输出字段定义 |
| 5 | `project/docs/一期术语与判断口径.md` | 15分钟 | 核心术语（延期/黄灯/结项申请等） |

**第一天阅读目标：理解原型能做什么 + 不能做什么 + 怎么跑起来。**

## 二、第一天跑什么

### 第一步：确认环境

```bash
# 确认 Python 环境
python3 --version

# 确认工作区路径
pwd
```

### 第二步：运行原型

```bash
cd /home/admin/.openclaw/workspace-digital-employee/project
python3 scripts/run_prototype.py
```

**预期结果：** 输出中 `通过：7/7`，末尾有领导摘要预览。

### 第三步：运行压力测试

```bash
python3 scripts/run_week4_stress.py
```

**预期结果：** `✅ Week 4 压力回归完成`，无报错。

### 第四步：查看最新输出

```bash
ls -t project/data/output/*.txt | head -3
```

打开其中一个领导摘要文件，确认内容完整。

## 四、第一天不要做什么

| # | 禁止事项 | 原因 |
|---|---------|------|
| 1 | 不得修改输入样本文件 | 样本是基准，修改后无法对比 |
| 2 | 不得修改已冻结的规则/术语 | 规则已锁定，改变会破坏判断一致性 |
| 3 | 不得在没有文档的情况下修改代码 | 先记录问题，再找文档依据 |
| 4 | 不得把样本数据说成真实数据 | 两者严格区分，当前原型使用固定样本 |
| 5 | 不得跳过阶段推进 | 阶段门控是项目节奏，不得跳过 |

---

## 五、第一天常见问题

| 问题 | 答案 |
|------|------|
| 运行报错 ImportError | 检查 `src/` 目录结构是否完整 |
| 7/7 不是全 PASS | 截图中发到群/邮件，等人排查 |
| 输出文件是空的 | 检查样本文件是否被破坏 |
| GitHub push 失败 | 已知问题，见 `docs/GITHUB_SYNC_ISSUE.md`，本地代码正常 |
| 不知道读哪个文档 | 先读 `RUNBOOK.md`，再读 `新Agent开工清单.md` |

## 六、第一天结束时的交付物

第一天结束时，你应当有：

```
□ 能运行 run_prototype.py 并得到 7/7 PASS
□ 能运行 run_week4_stress.py 并完整执行
□ 能在 project/data/output/ 找到最新的3个输出文件
□ 知道当前稳定基线版本是 c1e85bc
□ 知道 RUNBOOK.md 在哪里
□ 知道遇到问题去找哪个文档
□ 知道 GitHub push 失败是已知工程问题，不影响本地运行
```

## 七、第二天做什么

第二天起，可以开始以下工作（按优先级）：

1. 阅读 `project/docs/阶段推进总表.md`，了解当前阶段门控状态
2. 阅读 `project/docs/当前项目决策页.md`，了解已拍板决策
3. 运行一次真实数据替换（需要用户提供新样本）
4. 参与演示执行
