# 样本接入与验收规范

**版本：** v1.0
**状态：** 已定稿，待用户提供真实样本

---

## 一、样本分级定义

| 级别 | 名称 | 定义 | 支撑验证 | 不能支撑验证 |
|------|------|------|---------|------------|
| S0 | 模拟样本 | 人工构造，模拟真实场景 | 冒烟测试基础验证 | 真实场景覆盖 |
| S1 | 单份真实脱敏样本 | PMO提供，单项目单周期 | 口径校验、规则初验 | 跨周期场景、批量场景 |
| S2 | 多份真实脱敏样本 | PMO提供，≥3个项目或周期 | 原型开发验证、冒烟测试完整验证 | 连续运行场景 |
| S3 | 试点期连续样本 | 试点运行中持续产生，≥2个连续周期 | MVP收敛、演示版、运营验证 | - |

## 二、样本最小内容要求

### 2.1 周报/月报样本最小内容

| 字段 | 是否必须 | 最小要求 |
|------|---------|---------|
| project_name | ✅ 必须 | 项目全称 |
| owner | ✅ 必须 | 责任人姓名 |
| promised_items | ✅ 必须 | 包含至少1个时间节点或里程碑 |
| current_progress | ✅ 必须 | 有实质进展描述（非空） |
| risk_points | ⚠️ 可空 | 如有须包含风险关键词 |
| escalation_requested | ⚠️ 可空 | 默认false |
| report_date | ⚠️ 可空 | 如无，系统自动填当前日期 |
| report_type | ⚠️ 可空 | 默认weekly |

### 2.2 项目台账样本最小内容

| 字段 | 是否必须 | 最小要求 |
|------|---------|---------|
| project_name | ✅ 必须 | 项目全称 |
| owner | ✅ 必须 | 责任人姓名 |
| items | ✅ 必须 | 至少1个事项，每个事项含promised和current_progress |
| overall_risk_points | ⚠️ 可空 | 如有须包含风险关键词 |
| escalation_requested | ⚠️ 可空 | 默认false |
| last_updated | ⚠️ 可空 | 如无，系统自动填当前日期 |

## 三、样本命名规范

```
{类型}_{序号}_{格式类型}.json

类型：
  s0  - S0模拟样本
  s1  - S1单份真实脱敏样本
  s2  - S2多份真实脱敏样本
  s3  - S3试点期连续样本

格式类型：
  weekly  - 周报/月报格式（格式A）
  ledger  - 项目台账格式（格式B）

示例：
  s0_01_weekly.json    # S0模拟样本1，周报格式
  s1_01_weekly.json    # S1真实样本1，周报格式
  s2_01_ledger.json    # S2真实样本1，台账格式
  s2_02_weekly.json    # S2真实样本2，周报格式
  s3_01_weekly.json    # S3试点期样本1，周报格式
```

## 四、样本脱敏规范

### 4.1 必须脱敏

| 字段 | 脱敏规则 | 示例 |
|------|---------|------|
| project_name | 替换为"XX项目N"或"试点项目N" | "智慧园区一期" → "试点项目A" |
| owner | 替换为常见姓名 | "张明" → "张三" |
| 日期（绝对日期） | 替换为相对日期或保留格式擦除年份 | "2026-06-30" → "{基准日+X天}" |
| 公司/品牌名 | 替换为"某公司"等通用词 | "XX科技" → "某公司" |
| 具体地名 | 替换为"某地" | "北京" → "某地" |

### 4.2 不得脱敏（保持逻辑可验证性）

| 字段 | 原因 |
|------|------|
| promised_items的描述结构 | 判断逻辑依赖关键词存在 |
| current_progress的描述结构 | 判断逻辑依赖滞后词/完成词 |
| risk_points的描述结构 | 判断逻辑依赖风险关键词 |
| 事项的数量和粒度 | 影响事项归并逻辑 |
| 文字长度比例 | 影响无明确进展判断 |

### 4.3 脱敏执行方

| 样本级别 | 脱敏执行方 |
|---------|---------|
| S0 | Agent（人工构造，无敏感信息） |
| S1 | 用户/PMO（原始数据持有方） |
| S2 | 用户/PMO（原始数据持有方） |
| S3 | PMO持续脱敏后提交 |

## 五、样本验收标准

### 5.1 验收维度

| 维度 | 检查项 | 通过标准 |
|------|--------|---------|
| 字段完整性 | 必填字段是否存在 | 全部必填字段非空 |
| 时间信息完整性 | 时间节点是否可识别 | promised_items中可找到时间描述 |
| 状态信息可判定性 | 进展描述是否有判断关键词 | current_progress中含判断关键词（完成/滞后/正常等） |
| 责任人可识别性 | owner是否非空 | owner字段有内容 |
| 文字描述可解析性 | 描述是否为有效文字 | 文字>10字符，非纯符号 |

### 5.2 验收结果分类

| 结果 | 条件 | 处置方式 |
|------|------|---------|
| **通过** | 5个维度全部通过 | 直接进入对应样本库 |
| **部分通过** | 3-4个维度通过 | 降级使用，注明缺口；可用于局部测试 |
| **不通过** | <3个维度通过 | 退回补充；不进入任何测试链路 |

### 5.3 验收记录模板

```json
{
  "sample_id": "s1_01_weekly",
  "file": "s1_01_weekly.json",
  "submitted_date": "2026-04-07",
  "submitted_by": "[提供方姓名]",
  "dimensions": {
    "field_completeness": true,
    "time_info": true,
    "status_determinable": true,
    "owner_identifiable": true,
    "text_parseable": true
  },
  "result": "通过 | 部分通过 | 不通过",
  "gap_notes": "如有缺口，记录具体缺失内容",
  "disposition": "进入样本库 | 降级使用 | 退回补充"
}
```

## 六、样本缺口处理机制

| 缺口类型 | 判断标准 | 处理方式 |
|---------|---------|---------|
| 单个字段缺失（必填） | 字段为空或undefined | 退回补充；不得降级使用 |
| 字段内容模糊 | 文字<10字符，纯符号 | 退回补充 |
| 时间节点缺失 | promised_items无时间词 | 标记缺口，降级至S1；不用于判断规则验证 |
| 风险自述缺失 | risk_points为空 | 接受；标记为"无风险自述样本" |
| 描述无法解析 | 乱码或非目标语言 | 退回补充 |
| 部分维度不满足 | 3-4/5维度通过 | 降级使用；在报告中注明"降级样本，不具备全量验证能力" |

## 八、样本登记流程

```
用户提供样本
    │
    ▼
Agent执行验收检查（5个维度）
    │
    ├── 通过 → 写入 registry.json → 进入对应样本库
    ├── 部分通过 → 记录缺口 → 降级存放 → 通知用户
    └── 不通过 → 退回 → 用户补充后重新提交
```

**registry.json结构：**
```json
{
  "last_updated": "ISO8601",
  "samples": [
    {
      "id": "s1_01_weekly",
      "level": "S1",
      "file": "s1/s1_01_weekly.json",
      "result": "通过",
      "date_added": "2026-04-07",
      "added_by": "[用户名]"
    }
  ]
}
```
