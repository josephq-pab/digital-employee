## 三、形成的关键约束

**核心约束（3条必须遵守的铁律）：**

1. **人工审阅不可跳过**：系统输出必须经人工审阅方可作为正式依据，不得用于自动化决策。
2. **一期不接平台**：不接入OA、PMIS等系统；以人工填报周报/月报为唯一数据源。
3. **先验证后开发**：进入阶段4（原型的实现）前，必须完成样本数据校验和术语口径确认。

**进入开发阶段的5项前提条件（见《一期项目约束清单》）：**
- [ ] 术语口径文件获PMO确认
- [ ] 至少3份真实样例到位
- [ ] 输入规范确认
- [ ] 输出格式获管理层认可
- [ ] 冒烟测试用例全部定义

## 四、形成的关键规范

| 规范类型 | 核心内容 |
|---------|---------|
| **术语规范** | 8个核心术语的定义 + 判定标准，含"完成/延期/无明确进展/需领导介入"的判定边界 |
| **输入规范** | 6个必须字段（项目名/责任人/承诺事项/进展/风险点/是否请求介入），格式为结构化JSON |
| **输出规范** | 结构化JSON + 文本摘要，分离事实/判断/建议，附判断依据 |
| **判断规范** | 7条触发规则，覆盖低/中/高三级风险，可查表执行 |
| **变更规范** | 口径变更必须书面确认、同步测试用例、注明日期 |

## 五、已收敛的一期范围

**做：** 单项目周报/月报 → 结构化摘要，人工审阅后使用。

**不做什么（5条明确禁令）：**
- 不接OA/PMIS
- 不做自动催报/通知/升级
- 不做Dashboard/前端
- 不做多项目横向对比
- 不替代人工做升级决策

**暂不做什么：** 平台接入、多项目聚合、自动催报、图表展示。

**原型 vs MVP 差别已明确：**
- 原型：本地脚本 + 模拟样例，验证逻辑闭环
- MVP：API服务 + 真实样例，验证业务价值闭环
