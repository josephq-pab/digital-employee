# MVP进入条件与最小任务包

**版本：** v1.0
**目的：** 定义"什么时候可以进入MVP"以及"MVP最少要实现哪些功能"

---

## 一、MVP定义

> **MVP（最小可运行版本）**：在PMO真实试点中持续使用，具备基本可用性，可替代人工汇总至少一个环节的最小版本。

MVP是"能用"，演示版是"能看"。

---

## 二、MVP进入前必须满足的条件

| # | 条件 | 责任方 | 当前状态 |
|---|------|--------|---------|
| G1 | 阶段4冒烟测试A1-A7全部通过（100%） | 自动验证 | 🔴 未开始 |
| G2 | ≥3份真实样本（S2）到位且通过验收 | 用户/PMO | 🔴 未提供 |
| G3 | PMO确认术语口径在真实样本上校验通过 | PMO | 🔴 未确认 |
| G4 | PMO确认领导摘要格式符合使用要求 | PMO | 🔴 未确认 |
| G5 | 试点运行机制SOP已书面化并获PMO确认 | Agent（本文档） | 🟡 待完成（阶段3） |
| G6 | 管理层确认愿意审阅试点输出 | 管理层 | 🔴 未确认 |

**G1-G4全部满足后，方可进入阶段5。**

## 三、MVP最小能力集合

### 3.1 P0能力（必须具备，否则不是MVP）

| 能力 | 说明 | 验收方式 |
|------|------|---------|

| P0.5 错误可识别 | 输入异常时报错，不静默通过 | 边界测试用例验证 |

| 能力 | 说明 | 预计实现时机 |

## 五、实现前必须冻结的文档

以下文档在阶段4开发前必须冻结，**不得在开发期间修改**：

| 文档 | 冻结原因 |
|------|---------|
| 输入输出数据契约.md | 字段定义是开发的基准，改动代价极大 |
| 一期术语与判断口径.md | 口径是规则引擎的基准 |
| 判断规则与状态机说明.md | 规则逻辑是引擎的基准 |
| 冒烟测试计划与通过标准.md | 测试基准不得在执行中改 |

**冻结后的变更：** 必须提出变更申请，说明原因和影响，≥2人确认后方可变更。

## 六、允许在试点中迭代的内容

以下内容在试点运行中可以迭代优化，**不需要走冻结变更流程**：

| 内容 | 迭代条件 |
|------|---------|
| 规则权重/阈值 | 试点中出现明显误判，PMO确认 |
| 建议动作措辞 | 试点中PMO反馈措辞不当 |
| 演示场景 | 试点中发现更好的演示场景 |
| 规则补充（不影响核心框架） | PMO提出新场景，现有规则无法覆盖 |

## 七、MVP成功标准

**阶段5结束时，必须同时满足：**

| 标准 | 验证方式 |
|------|---------|
| PMO连续4周使用系统输出 | 周度记录可查 |
| PMO书面反馈"接受"或"可接受，需改进" | 书面文件存档 |
| 规则误判率<20% | 问题登记/总判断数 |
| 管理层连续2个月审阅 | 审阅记录可查 |
| P0.1-P0.5全部实现 | 逐项验收 |

## 八、MVP不是最终产品

| MVP包含 | MVP不包含（未来） |
|---------|-----------------|
| 本地脚本运行 | HTTP API服务 |
| 手动JSON输入 | 飞书/企微消息通道 |
| 手动存档 | 数据库持久化 |
| 单项目串行 | 多项目批量 |
| 无前端 | Dashboard/界面 |
| 无SLA | 稳定可用保证 |
