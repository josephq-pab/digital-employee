## 归因1：黄色预警未触发中风险判断

**严重程度：** 中
**编号：** QUAL-01

### 现象

4个黄灯事项（汽车行业×3期，学校行业×1期）的engine.py判断结果均为"低风险"。

### 根因分析

engine.py的`_judge_risk`方法中，黄灯alert的raw文本（如"项目逾期"）未被任何LAG_PATTERNS或RISK_SELF_REPORTED_PATTERNS匹配，导致：
- risk_signals = []
- status_signals = []
- 最终判断：低风险

### 修复方案

**方案A（适配层修复）：**
在`normalizer.py`的`build_normalized_item`中，当`is_yellow_alert=True`时，在risk_points字段中显式添加"[黄灯]"，使规则引擎能识别：

```python
# 在build_normalized_item中
if is_yellow_alert and not risk_points:
    risk_points = "[黄灯]存在逾期风险"
```

**方案B（规则层修复）：**
在`engine.py`中新增黄灯判断逻辑：

```python
# 在_judge_risk中
if 'is_yellow_alert' in inp and inp['is_yellow_alert']:
    return '中', False, None
```

**推荐：** 方案A（适配层），符合"适配层处理特殊信号，规则引擎保持纯净"原则

### 影响范围

- 领导摘要中的"延期/黄灯事项"分类
- 风险清单中的风险等级
- 事项总台账中的风险等级

## 归因2：结项申请进入风险清单

**严重程度：** 中
**编号：** QUAL-02

### 现象

7个结项申请（closure_requests）出现在风险清单的"其他风险"中，标记为"无明显风险信号"。

### 根因分析

`risk_list_generator.py`中的`is_risk`判断条件：

```python
is_risk = (
    item.get('is_yellow_alert')
    or risk_level in ('中', '高')
    or escalation
    or '延期' in item['project_name']
    or item.get('is_closure_request')  # ← 这里把结项申请也当作风险
)
```

`is_closure_request=True`本应是"风险已释放"的意思，但被错误地当作"有风险"。

### 修复方案

在`risk_list_generator.py`中修改判断逻辑：

```python
# 结项申请 = 风险已释放，不进入风险清单
if item.get('is_closure_request'):
    continue  # 不进入风险清单
```

### 影响范围

- 风险清单中的数量（当前11项，修复后约4项）
- 领导摘要中的"已完成/已解决"分类（不受影响）

## 归因3：行业维度与项目维度的结构性差异导致匹配率低

**严重程度：** 中（但这是数据本身的问题，非代码问题）
**编号：** QUAL-03

### 现象

12条映射中，8条unmatched（67%），其中大部分是行业维度（汽车行业/学校行业）无法匹配到台账中的项目。

### 根因分析

- 周报按行业维度组织（如"汽车行业"是独立条目）
- 台账按项目维度组织（如"六策合一-汽车零部件"是子项）
- 两者维度不同，属正常差异

### 修复方案

需PMO确认MAP-01~04的映射关系：
- MAP-01：汽车行业 → ledger-10（六策合一-汽车零部件）
- MAP-02：学校行业 → 待确认
- MAP-03：3个结项申请 → 确认是否已结项

### 当前 workaround

适配层使用fuzzy匹配，当前4/12已正确匹配（33%）。其余8条已标注"unmatched"，不影响核心判断逻辑。

## 归因4：事项重复出现在多期周报中

**严重程度：** 低
**编号：** QUAL-04

### 现象

"汽车行业"在W50/W51/W52三期周报中均出现，被当作3个独立事项处理。

### 根因分析

当前归一化逻辑按"周报→事项"处理，未做跨期去重。

### 修复方案（可选，阶段5再处理）

在`run_prototype.py`中增加去重逻辑：

```python
# 按事项名称+期间去重
seen = set()
unique_normalized = []
for item in all_normalized:
    key = (item['project_name'][:20], item['period'])
    if key not in seen:
        seen.add(key)
        unique_normalized.append(item)
```

## 归因5：责任人字段为空

**严重程度：** 低
**编号：** QUAL-05

### 现象

所有11个事项的"责任人"字段均显示"[待关联]"。

### 根因分析

周报本身无owner字段，适配层未能有效关联台账的owner_department。

### 当前状态

已实现台账匹配（fuzzy），但匹配后owner_department未写入归一化item。

### 修复方案

在`normalizer.py`中，匹配到ledger_record后，提取其owner_department字段：

```python
# 在normalize_weekly_report中
if ledger_rec:
    item['owner'] = ledger_rec.get('owner_department', '') or '[待关联]'
```
