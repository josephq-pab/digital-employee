# 演示后分支处理说明

**版本：** v1.0
**日期：** 2026-04-07
**用途：** 预先设计两条分支，避免演示后再次失速

---

## 一、分支总览

```
演示完成
    ↓
收集反馈
    ↓
判断分支
    ↓
├── 分支A：演示通过
│         ↓
│         推进Q01/Q03/Q04确认
│         ↓
│         进入MVP试跑准备
│
└── 分支B：演示有较大反馈
          ↓
          先修订，再继续
          ↓
          分支B1：规则修订 → 等确认 → 再演示
          分支B2：展示修订 → 迭代演示 → 再评估
          分支B3：数据修订 → 补充样本 → 再评估
```

---

## 分支A：演示通过

**触发条件（满足任一即可）：**
1. 无实质性负面反馈
2. 反馈均为展示问题（P2级别）
3. 反馈均为需求扩展（一期不做）
4. PMO/管理层口头认可，同意推进

**最小动作集：**

```
A1. 记录反馈（30分钟）
    → 填写演示版验收记录表.md
    → 所有反馈归类到演示反馈归并规则.md

A2. 更新门控状态（如需）（15分钟）
    → 基于口头认可更新G1-G7（如有）
    → 明确说明"口头认可，待正式签字"

A3. 发出确认包（用户负责）（1天）
    → 管理层确认外发版 → 管理层签字
    → PMO确认外发版 → PMO签字
    → 试点范围确认外发版 → PMO确认

A4. 推进Q01/Q03/Q04（等待外部）
    → 按确认链日更新SOP.md每日跟踪
    → 收到确认 → 更新门控 → 进入MVP

A5. 准备MVP试跑（并行）
    → 按演示后到MVP试跑准备路径.md执行
    → 准备试点样本
    → 制定试跑SOP
```

**退出条件：** Q01+Q03+Q04全部确认 → 进入MVP试跑准备

---

## 分支B：演示有较大反馈

**触发条件（满足任一即可）：**
1. PMO/管理层对核心规则有异议（涉及PMO-01/02）
2. 反馈涉及"这个判断我无法接受"（影响核心判断逻辑）
3. 输出格式被明确拒绝（MGT-01）
4. 数据质量被质疑（样本本身有问题）

**分支B1：规则修订**

**触发条件：** 反馈涉及Q01/Q03（术语口径/输出格式），PMO/管理层要求修改规则

```
B1.1 记录规则反馈（30分钟）
     → 写入规则问题待确认清单.md
     → 标注"涉及Q01/Q03"

B1.2 判断是否需要修改engine.py（1小时）
     → 分析：当前provisional规则是否需要调整
     → 结论：调整/维持

B1.3 发出修订版确认包（用户负责）（1天）
     → 更新确认包内容
     → 重新发出

B1.4 等PMO/管理层重新确认（等待外部）
     → 收到确认 → 更新规则 → 更新门控

B1.5 重新演示或直接进入MVP（取决于反馈严重度）
     → 如重大规则变化 → 重新演示
     → 如轻微调整 → 直接进入MVP
```

**退出条件：** Q01/Q03重新确认 → 更新规则 → 进入MVP

---

## 分支B2：展示修订

**触发条件：** 反馈均为展示层（格式/标签/顺序），不涉及核心规则

```
B2.1 记录展示反馈（30分钟）
     → 写入展示问题清单.md

B2.2 识别展示修改范围（1小时）
     → 判断：哪些可以在下次演示前修改
     → 判断：哪些需要等确认后才能修改

B2.3 执行展示修改（Agent自行处理）（1天）
     → 修改summary_generator.py
     → 修改ledger_generator.py
     → 修改risk_list_generator.py

B2.4 回归测试（1小时）
     → python project/scripts/run_prototype.py
     → 确认输出正常

B2.5 迭代演示（安排下次演示）
     → 执行新演示
     → 收集新反馈
     → 重新评估分支
```

**退出条件：** 展示问题全部修复 → 重新演示 → 分支A或B2

---

## 分支B3：数据修订

**触发条件：** 反馈涉及样本数据质量问题（映射错误/owner错误/数据错误）

```
B3.1 记录数据反馈（30分钟）
     → 写入数据问题反馈表.md

B3.2 判断数据问题范围（1小时）
     → 分析：是个别问题还是系统性错误
     → 判断：是否需要重新准备样本

B3.3 联系数据提供方（用户负责）（1-3天）
     → 将反馈传达给PMO/数据提供方
     → 请求重新提供/修正样本

B3.4 补充样本到位后更新S2（Agent处理）（1天）
     → 重新验收样本
     → 更新registry
     → 更新门控G5

B3.5 重新运行prototype（1小时）
     → python project/scripts/run_prototype.py
     → 确认输出改善

B3.6 重新演示或直接评估（视反馈严重度）
```

**退出条件：** 数据问题修复 → 重新演示 → 分支A

---

## 三、分支决策树

```
演示后
    ↓
有实质性负面反馈？
    ├── 无 → 分支A（演示通过）
    └── 有 → 反馈涉及核心规则/格式？
              ├── 否（展示/数据/扩展）→ 分支B2/B3
              └── 是（涉及Q01/Q03）→ 分支B1
                                      ↓
                              需要修改engine.py？
                              ├── 是 → 修改→等确认→重新演示
                              └── 否 → 等确认→进入MVP
```

## 四、各分支最短时间估算

| 分支 | 最短时间 | 说明 |
|------|---------|------|
| 分支A | 7-14天 | Q01/Q03/Q04确认时间 |
| 分支B1（规则修订） | 14-21天 | 规则修订+等确认+重新演示 |
| 分支B2（展示修订） | 3-5天 | 修改+回归+迭代演示 |
| 分支B3（数据修订） | 7-14天 | 数据方修正+重新验收 |

---

## 五、分支A/B的核心区别

| 项目 | 分支A | 分支B |
|------|-------|-------|
| 触发条件 | 无实质性负面反馈 | 有影响核心的反馈 |
| 是否修订规则 | 否 | 可能 |
| 是否重新演示 | 否 | 可能 |
| MVP进入速度 | 快 | 慢 |
| 外部依赖 | Q01/Q03/Q04确认 | 确认+可能的规则修订 |
