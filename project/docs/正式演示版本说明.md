## 二、本版修复内容

### QUAL-01：黄灯判断修复

| 项目 | 内容 |
|------|------|
| 问题 | 黄灯事项被判断为"低风险" |
| 根因 | 适配层normalizer.py向risk_points注入信号，但engine.py的LAG_PATTERNS检查current_progress |
| 修复 | 向current_progress字段注入"进度滞后"文本 |
| 修复文件 | project/src/adapters/normalizer.py |
| 修复后效果 | 黄灯→中风险，状态→关注 |

### QUAL-02：结项申请排除

| 项目 | 内容 |
|------|------|
| 问题 | 7个结项申请误入风险清单 |
| 根因 | risk_list_generator.py将is_closure_request纳入is_risk判断 |
| 修复 | 对is_closure_request=True的事项直接continue跳过 |
| 修复文件 | project/src/generators/risk_list_generator.py |
| 修复后效果 | 风险清单从11项→4项（仅黄灯） |

### QUAL-05：owner字段回填

| 项目 | 内容 |
|------|------|
| 问题 | 所有事项的owner显示"[待关联]" |
| 根因 | 适配层处理closure_requests时未传递ledger_record |
| 修复 | 为closure项也查找映射并传递ledger_record |
| 修复文件 | project/src/adapters/normalizer.py |
| 修复后效果 | 有映射事项的owner正确回填；无映射事项仍为"[待关联]" |

## 七、演示时边界话术（参考）

**开场必须说：**
```
"这是条件性实现版本。判断规则尚未被PMO正式确认，
输出格式尚未被管理层正式确认。
我们用这个版本做演示，是为了收集反馈，
是为了让正式版本更准确。"
```

**被问及与正式版差别时：**
```
"正式版需要PMO确认术语口径、管理层确认输出格式、
确认试点范围。这三个确认收到后，
我们会把provisional规则升级为正式版本，
然后进入MVP试点。"
```
