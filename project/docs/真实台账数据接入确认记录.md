## 一、样本JSON格式基准（供参考对比）

**文件路径：** `samples/samples/project_ledger/project_ledger_latest3.json`

**JSON结构：**
```json
{
  "sample_type": "project_ledger",
  "sample_version": "v1",
  "is_desensitized": true,
  "source_workbook": "项目汇总.xlsx",
  "selected_periods": ["1222-1226", "1215-1219", "1208-1212"],
  "records": [
    {
      "record_id": "ledger-31",
      "worksheet": "主表",
      "row_number": 31,
      "sequence": "1.3.2",
      "project_type": "重点项目",
      "project_name": "制定贷款促投放提升方案，有序压降票证规模",
      "record_type": "主体",
      "objective_or_requirement": "...",
      "owner_department": "某分行",
      "due_date": "2025-12-31",
      "progress_by_period": {
        "1222-1226": "1、“上一横”，进一步补充交付物细节；\n2、“中一竖”，本周为第四周，整体进展符合预期；\n3、“下一横”，项目收尾中。",
        "1215-1219": "1、“上一横”，完成最终交付物；\n2、“中一竖”，已明确方案，本周为第3周。",
        "1208-1212": "1、“上一横”，完成最终交付物；\n2、“中一竖”，已明确方案。。"
      },
      "subitems": [
        {
          "subitem_id": "ledger-31-1",
          "content": "子事项内容",
          "status": "进行中"
        }
      ]
    }
  ]
}
```

**重要说明：台账status字段的实际情况**

> 台账中的 `progress_by_period` 字段存储的是**文本描述**（各期进展说明），而非标准化的green/yellow/red状态灯。
> 系统根据**周报数据**（key_alerts/closure_requests等）来判断结论卡状态（黄灯/延期/结项），而非根据台账status字段。
> 因此台账不需要green/yellow/red标准化的status字段。

**必要字段（系统运行所依赖）：**

| 字段路径 | 类型 | 说明 | 备注 |
|---------|------|------|------|
| `records[].record_id` | string | 台账记录唯一ID | ❌ 不可缺 |
| `records[].project_name` | string | 项目名称 | ❌ 不可缺 |
| `records[].progress_by_period` | object | 各期进展文本 | ✅ 文本即可 |
| `records[].subitems` | array | 子事项列表 | ✅ 可空 |
| `records[].owner_department` | string | 负责部门 | ✅ 可缺 |

## 三、重要说明：台账status值不需要标准化

**⚠️ 重要发现：台账不需要green/yellow/red标准化的status字段**

系统结论卡（黄灯/结项/延期/正常）的判断依据是**周报数据**，不是台账status字段。

因此：
- 台账的 progress_by_period 是文本描述，不影响系统判断
- 不需要在接入前把台账status值标准化为green/yellow/red
- 台账数据接入只需确保 records 结构和 project_name 字段存在
