# 原型_冒烟测试_演示版_MVP关系说明

**版本：** v1.0
**目的：** 消除概念混淆，为后续阶段提供明确的口径基准

---

## 一、四个阶段的关系

```
【阶段4】原型实现
    目标：逻辑闭环可跑通
    输入：模拟/脱敏样例
    验证：冒烟测试100%通过
    ↓ 前提：PMO确认 + ≥3份真实样本到位
【阶段5】演示版
    目标：业务价值可讲可看
    输入：真实样本（≥3份）
    验证：PMO/管理层审阅反馈
    ↓ 前提：PMO书面反馈 + MVP范围确定
【阶段5末】MVP收敛
    目标：真实试点中可用
    验证：PMO实际使用 + 书面反馈
    ↓
【未来】生产级
    目标：稳定运行
    新增：API服务、飞书/企微接入、持久化
```

## 二、原型（阶段4产出）

### 2.1 定义

> 用本地脚本+JSON实现核心逻辑，验证"规则推理→判断→输出"这条链路可以跑通。

### 2.2 必须具备的特征

| 特征 | 说明 |
|------|------|
| 本地可运行 | python3 engine.py input.json → output.json |
| JSON输入/输出 | 不做前端、不做界面 |
| 5条核心规则 | TC01-TC05全部通过 |
| 可演示 | bash run_dev.sh 能展示完整闭环 |
| 有代码结构 | 按模块拆分，不是单文件全写 |

### 2.3 不具备的特征（原型做不到的事）

| 特征 | 说明 |
|------|------|
| 真实数据验证 | 没有真实PMO样本，只用模拟数据 |
| 格式B（台账）支持 | 阶段5才支持 |
| PMO反馈 | 无实际用户反馈 |
| 业务价值证明 | 只证明逻辑对，不证明有用 |
| 稳定性验证 | 无压力测试、边界测试（扩展用例） |

### 2.4 成功标准

- run_smoke.sh 通过（TC01-TC05）
- bash run_dev.sh 可展示完整输出
- 代码按模块拆分，可被其他人维护

## 三、冒烟测试（原型验证手段）

### 3.1 定义

> 用5条核心测试用例验证核心规则在已知场景下是否按预期工作。

### 3.2 冒烟测试验证什么

| 验证项 | 覆盖场景 |
|--------|---------|
| 正常无风险 | TC01 |
| 滞后+风险自述 | TC02 |
| escalation请求升级 | TC03 |
| 严重滞后 | TC04 |
| 无风险正常进度 | TC05 |

### 3.3 冒烟测试不验证什么

| 未验证项 | 说明 |
|---------|------|
| 真实样本处理 | 只用模拟数据 |
| 格式B（台账） | TC未覆盖 |
| 边界条件 | 文本超长、空字段降级等 |
| 稳定性 | 无压力测试 |
| PMO满意度 | 无用户反馈 |

### 3.4 冒烟测试通过≠原型完成

冒烟测试通过是原型的**必要条件**，不是**充分条件**。

原型完成还必须：
- [ ] 演示可运行（bash run_dev.sh）
- [ ] 代码按模块拆分
- [ ] 输出格式符合OUTPUT_SCHEMA
- [ ] 判断依据完整（reasoning字段非空）

## 四、演示版（阶段5产出）

### 4.1 定义

> 用真实样本数据向PMO/管理层展示：给定真实项目数据，系统能生成什么样的摘要，并收集反馈。

### 4.2 必须具备的特征

| 特征 | 说明 |
|------|------|
| 真实样本 | PMO提供的≥3份脱敏真实数据 |
| 领导摘要可读 | 500字以内，可打印/投影 |
| PMO审阅 | PMO负责人或管理层实际审阅 |
| 书面反馈 | PMO给出"接受/不接受/需调整" |

### 4.3 演示版不是MVP

| 对比项 | 演示版 | MVP |
|--------|--------|-----|
| 目标 | 验证业务价值可讲可看 | 验证业务价值可运行可用 |
| 数据 | 真实样本 | 真实试点数据 |
| 使用方式 | 一次性演示 | 持续使用 |
| PMO反馈 | 口头/书面均可 | 必须书面 |
| 格式B支持 | 可选 | 必须 |
| 稳定性要求 | 无 | 基本可用 |

### 4.4 演示版成功标准

- [ ] PMO用真实样本跑通了系统
- [ ] 管理层审阅了摘要输出
- [ ] PMO给出了"接受/需调整"反馈
- [ ] 反馈中无"完全不可用"类拒绝

## 五、MVP（阶段5末收敛）

### 5.1 定义

> 在PMO真实试点中持续使用的最小可用版本，具备基本可用性，可替代人工汇总的至少一个环节。

### 5.2 MVP必须具备的能力

| 能力 | 说明 |
|------|------|
| 真实数据处理 | 格式A（周报/月报）+ 格式B（台账） |
| 领导摘要生成 | 符合管理层需求 |
| 事项台账输出 | 至少可导出存档 |
| 风险清单输出 | 至少可识别并列出 |
| PMO持续使用 | 连续2个周期以上使用 |
| PMO书面接受 | "接受使用"或"可接受，需改进" |

### 5.3 MVP不是最终产品

| 区别 | MVP | 未来生产级 |
|------|-----|---------|
| API服务 | 可选（手动运行亦可） | 必须HTTP API |
| 消息接入 | 无 | 飞书/企微消息通道 |
| 持久化 | 无（JSON存档） | 数据库 |
| 多项目 | 单项目串行 | 批量多项目 |
| 前端 | 无 | Dashboard/界面 |
| SLA | 无 | 稳定可用保证 |

## 六、概念边界总结

| 概念 | 核心差异 | 验证方式 |
|------|---------|---------|
| **原型** | 逻辑可跑 | 冒烟测试通过 |
| **冒烟测试** | 5条核心规则正确 | run_smoke.sh |
| **演示版** | 业务价值可讲可看 | PMO审阅反馈 |
| **MVP** | 真实可用可运行 | PMO书面接受+持续使用 |

**核心区别一句话：**
- 原型：逻辑对了（能跑）
- 演示版：看着有用（能讲）
- MVP：真的能用（能用）
