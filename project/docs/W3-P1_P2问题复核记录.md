## 一、W3-P1 复核：结项申请内容误合并

### 1.1 具体表现

**触发样本**：1103-1107 alert_lines（来源：normalized weekly report）

**原始 alert_lines 片段**：
```
"2、搭建战略客户样板和北京分行战客专班建设，战客部预计本周内邮件汇报，但目前尚未汇报。项目逾期，黄灯提示。"
```

**期望行为**：
- "搭建战略客户样板" → closure_requests（申请结项）
- "北京分行战客专班建设" → key_alerts（黄灯提示）

**实际行为**：
- `adapt_alert_line_to_key_alert()` 提取 project_name = "2、搭建战略客户样板和北京分行战客专班建设"（整句）
- 经正则 `r'^([^，,：:]+)'` 匹配，项目名为 "2、搭建战略客户样板和北京分行战客专班建设"
- 后续在 `build_normalized_item` 中注入"进度滞后"信号
- 输出中此项为 "2、搭建战略客户样板和北京分行战客专班建设"，项目名包含不应出现的内容

### 1.2 根因分析

**根因分类**：⚠️ 数据质量/样本标注问题 + 适配层健壮性不足

1103-1107 alert_lines 中混入了 request_lines 的内容（"搭建战略客户样板和北京分行战客专班建设"整体是一句包含结项申请的话）。这是样本预处理阶段的数据标注问题，非原型引擎 bug。

**适配层不足**：即使面对上述数据，适配层也应更智能地处理"和"连接的并列结构。

### 1.3 影响范围

| 影响维度 | 程度 | 说明 |
|---------|------|------|
| 黄灯识别 | 轻微 | "北京分行战客专班建设"被正确识别为黄灯，只是项目名不干净 |
| 结项识别 | 轻微 | "搭建战略客户样板"结项申请被误合并入黄灯，而非在 closure_requests 中出现 |
| 台账匹配 | 无 | 台账中此项目被匹配（1103-1107_北京分行战客专班建设） |
| 输出口径 | 无 | 不影响三类输出结构完整性 |

**严重度确认**：P2（适配层/数据质量问题，不影响核心引擎）

## 二、W3-P2 复核：延期申请未单独识别

### 2.1 具体表现

**触发样本**：1117-1121

**原始 request_lines**：
```json
["一、一个项目申请延期，请您审批："]
```

**原始 raw_text 片段**：
```
一、一个项目申请延期，请您审批：
"数字经营"规划推进：原计划11月末完成，因监管检查、新核心切换等原因，结现部申请项目延期至12月上旬。
```

**期望行为**：
- 延期申请 → 识别为 extension_requests，project_name = "数字经营"规划推进

**实际行为**：
- `adapt_request_lines_to_closure_requests()` 中 `if line.startswith('一、')` 跳过标题行
- 没有后续内容行，延期申请完全丢失
- "数字经营"规划推进仅作为 management_summary.delayed_items 落入黄灯信号

### 2.2 根因分析

**根因分类**：✅ 适配层实现问题

`adapt_request_lines_to_closure_requests()` 只处理 request_lines 数组中的内容行，但 1117-1121 的 request_lines 只有标题行（无内容行）。真正的延期申请内容在 raw_text 中。

适配层未实现"标题-内容配对"逻辑：当 request_lines 中只有标题时，应从 raw_text 中提取后续内容。

### 2.3 影响范围

| 影响维度 | 程度 | 说明 |
|---------|------|------|
| 延期申请识别 | 100% 丢失 | "数字经营"延期申请完全未被识别为独立事项类型 |
| 黄灯识别 | 不影响 | 延期申请落入 delayed_items，仍有风险信号 |
| 输出口径 | 轻微 | 领导摘要/风险清单中无"延期申请"类别，只有黄灯 |

**严重度确认**：P2（适配层实现问题，不影响核心引擎）
