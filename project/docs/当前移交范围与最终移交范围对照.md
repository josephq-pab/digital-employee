# 当前移交范围与最终移交范围对照

> 版本：v1.0（2026-04-16）
> 用途：明确"当前已有什么"vs"最终还需要什么"

---

## 一、核心对照

| 维度 | 当前已具备 | 最终移交前必须补齐 |
|------|-----------|------------------|
| 主流程 | 端到端可运行，但口径仍在修正 | 主流程冻结，数据口径稳定 |
| 代码质量 | 原型级代码，可演示 | 生产级代码，基本无P0/P1缺陷 |
| 页面/MVP | MVP页面可访问，表单可提交 | 页面体验收口，表单逻辑锁定 |
| 数据口径 | 有样本数据，口径规则存在 | DATA-001登记的口径问题全部关闭 |
| 连续运行 | 样本验证通过，生产环境未测 | S1 连续3期无P0，影子运行验证 |
| 业务确认 | 无业务方正式反馈 | B1~B4 业务可接受性全部确认 |
| 文档 | 开发基线文档齐全 | 冻结版操作手册+生产SOP |
| 人员 | 当前Agent可维护 | 新接手方能独立运维 |

---

## 二、当前移交 = 开发基线可接手

### 当前已具备的移交内容

**代码层面：**
- 核心 Python 代码（adapters/generators/loaders）
- 入口脚本（run_prototype.py / run_week4_stress.py）
- 样本数据（周报/台账/映射 JSON）

**文档层面：**
- RUNBOOK.md（操作手册）
- 演示运行说明/演示检查清单
- 新接手者首日操作清单
- 正式交付结论落版.md（当前版本定位说明）

**验证层面：**
- 7/7 冒烟测试 PASS
- 压力测试 PASS（38条/9期）
- 正式演示验证 PASS
- 正式交接验证 PASS

### 当前移交的真实含义

**"当前移交"的意思是：**
- 开发基线已稳定，可以交给别人继续完善
- 不是"最终版"，是"开发基线版"
- 新接手者可以在此基础上继续推进冻结版建设

---

## 三、最终移交 = 冻结版完成后移交

### 最终移交前必须完成的内容

**主流程冻结（当前：待冻结 → 最终：已冻结）：**
- 数据口径修正完成（DATA-001 全部关闭）
- 归一化规则稳定
- 规则判断逻辑稳定
- 输出格式和字段定义稳定

**体验收口（当前：MVP级 → 最终：生产级）：**
- 页面体验完成
- 输出文件格式锁定
- MVP表单逻辑锁定

**连续运行验证（当前：样本验证 → 最终：生产验证）：**
- S1 连续3期无P0
- 影子运行完成
- 无P0/P1阻断性问题

**业务可接受性（当前：0确认 → 最终：4项确认）：**
- B1~B4 业务方全部确认

---

## 四、迁移后与迁移前的职责对比

| 阶段 | 主责 | 任务 |
|------|------|------|
| **迁移前（当前→冻结版）** | 当前Agent | 完善主流程、修正口径、验证连续运行、收口体验 |
| **迁移后（冻结版→生产）** | 新接手方 | 替换模型、替换环境、接生产数据、少量配置/缺陷 |

---

## 五、迁移决策门控

**在达到以下条件之前，禁止启动最终迁移：**

| 门控 | 当前状态 | 解锁条件 |
|------|---------|---------|
| 主流程冻结 | 未冻结 | DATA-001 全部关闭 + 规则逻辑稳定 |
| 体验收口 | 未收口 | MVP页面冻结 + 输出格式稳定 |
| 连续运行验证 | 未验证 | S1 连续3期无P0 |
| 业务确认 | 0/4 | B1~B4 至少3/4 确认 |
