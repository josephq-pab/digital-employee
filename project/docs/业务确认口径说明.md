# 业务确认口径说明

> 版本：v1.0（2026-04-16）
> 阶段：冻结建设版主线 → 阶段6L
> 用途：统一B1~B4业务确认的判断标准，避免临时变更口径

---

## 一、B1~B4 各自通过标准

### B1：业务方试用反馈无E类（阻断）问题

| 字段 | 内容 |
|------|------|
| **通过标准** | 收到≥1条有效反馈，且反馈中无E类（阻断）问题描述 |
| **有效反馈定义** | 通过MVP页面内ABC决策表单提交的具体反馈，包含：ABC决策结论 + 问题描述（B/C选项时必填） |
| **E类（阻断）问题定义** | 系统完全不可用 / 输出完全不符合业务场景 / 存在数据安全/合规问题 |
| **证据形式** | MVP反馈表单提交记录（含ABC选择 + 问题描述） |
| **通过条件** | 反馈中ABC选择为A，或B/C选择但问题描述为非阻断性（属于P2类优化建议） |

### B2：核心用户（PMO/管理层）试用意愿

| 字段 | 内容 |
|------|------|
| **通过标准** | PMO或管理层核心用户明确表示愿意将数字员工纳入并行运行流程 |
| **明确表态定义** | 书面（邮件/消息截图）或口头（有记录）的"愿意使用"或"可以推进" |
| **模糊表态（非通过）** | "还行吧"/"可以看看"/"先这样"/"没什么问题"等无法确认意愿的表述 |
| **证据形式** | 邮件/消息截图 + 对方姓名+职级+日期 |
| **通过条件** | 有具体人+具体意愿表示+日期的记录 |

### B3：结论卡准确率获业务方认可

| 字段 | 内容 |
|------|------|
| **通过标准** | 业务方确认：结论卡的判断（哪些是黄灯/结项/延期）准确率可接受 |
| **认可定义** | 口头或书面表示"结论卡判断基本准确"或"可以作为参考" |
| **不认可定义** | 指出具体判断错误（如"这个应该是黄灯不是绿灯"）且理由充分 |
| **证据形式** | 口头（有记录）或书面确认 |
| **通过条件** | 有具体人+对结论卡准确性的明确认可+日期 |

### B4：数字员工输出可作为管理会议辅助材料

| 字段 | 内容 |
|------|------|
| **通过标准** | 业务方确认：领导摘要、台账、风险清单可以作为管理会议的辅助参考材料 |
| **辅助材料定义** | 在管理会议场景下，数字员工输出与人工周报并列作为参考 |
| **认可定义** | 口头或书面表示"可以作为辅助材料"或"可以用" |
| **证据形式** | 口头（有记录）或书面确认 |
| **通过条件** | 有具体人+对输出可用性的明确认可+日期 |

---

## 二、"有效反馈"定义

**有效反馈必须包含：**

1. **ABC决策结论**（A/B/C三选一）
2. **问题描述**（B或C选择时必填，A选择时选填）
3. **来源可追溯**（谁提交的）

**无效反馈（不算入B1）：**
- 抽象评价（"挺好的""还不错"）
- 口头态度（无记录）
- 非反馈性言论（"了解""知道了"）

---

## 三、"认可"的证据强度排序

**从强到弱：**

| 证据强度 | 形式 | 示例 |
|---------|------|------|
| 最强 | 书面+签名 | 邮件/审批/书面确认函 |
| 强 | 消息+身份 | 微信/钉钉截图+对方姓名+职级 |
| 中 | 口头+记录 | 会议纪要中记录"XX表示认可" |
| 弱 | 口头+无记录 | 无法追溯的口头表态 |
| 无效 | 模糊表态 | "还行""可以吧""差不多" |

**弱证据需要至少2个独立来源方可视为通过。**

---

## 四、B1~B4证据记录规范

每项确认必须记录：

```
## B[X] 确认记录

- **确认人**：姓名 / 职级 / 部门
- **确认时间**：YYYY-MM-DD
- **确认方式**：书面/口头（有记录）/口头（无记录）
- **确认内容**：具体说了什么/提交了什么
- **证据来源**：截图/录音/会议纪要/邮件
- **是否符合通过标准**：是/否
- **是否符合证据强度要求**：是/否
- **结论**：✅ 通过 / ⏳ 待补充证据 / ❌ 未通过
```

---

## 五、禁止事项

1. **不得**把"0条反馈"写成"业务已通过"
2. **不得**把模糊表态（"还行吧"）算作B2/B3/B4通过
3. **不得**把口头+无记录的表态作为唯一证据通过B2/B3/B4
4. **不得**事后补编证据
5. **不得**在标准模糊时临时发明新标准
