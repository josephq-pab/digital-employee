## 一、当前样本映射数据（参考基准）

**样本路径：** `samples/samples/linked/weekly_to_ledger_mapping_latest3.json`

**数据结构：**
```json
[
  {
    "period": "1222-1226",
    "source_type": "closure_requests",
    "source_project_name": "制定贷款促投放提升方案",
    "match_type": "fuzzy",
    "ledger_record_id": "ledger-31",
    "ledger_project_name": "制定贷款促投放提升方案，有序压降票证规模"
  }
]
```

**字段说明：**
| 字段 | 说明 |
|------|------|
| period | 周报期数 |
| source_type | 来源类型：closure_requests/new_items/key_requests等 |
| source_project_name | 周报中的事项名称 |
| match_type | 匹配类型：fuzzy（模糊匹配）/ exact（精确匹配）/ no_match |
| ledger_record_id | 匹配到的台账record_id |
| ledger_project_name | 台账中的项目名称 |

## 二、映射责任矩阵（谁负责什么）

### 2.1 系统自动完成（开发方负责）

| # | 系统自动完成的事项 | 说明 |
|---|------------------|------|
| 1 | 周报事项 → 台账项目fuzzy匹配 | 使用现有find_ledger_record逻辑 |
| 2 | 匹配结果记录（成功/失败） | 输出未匹配项目清单 |
| 3 | 映射关系存储到mapping JSON | 用于后续追溯 |
| 4 | 匹配失败项标注为"no_match" | 供人工审查 |

**系统自动映射的能力：**
- 按项目名称fuzzy匹配（周报事项 ↔ 台账项目）
- 匹配成功 → 返回台账记录
- 匹配失败 → 记录为"未匹配"（no_match）

**系统自动映射的限制：**
- 项目名称必须有规律性
- 名称差异过大会导致匹配失败
- 多对一情况可能需要人工判断

### 2.2 项目方需补什么（项目方负责）

| # | 项目方负责事项 | 说明 |
|---|--------------|------|
| 1 | fuzzy匹配失败项的人工补充映射 | 补充 no_match 项的 ledger_record_id |
| 2 | 若有现成映射JSON，提供文件 | 跳过系统自动匹配 |
| 3 | 确认台账项目名称与周报事项名称的对应关系 | 业务层面的确认 |
| 4 | 评估fuzzy匹配的准确率 | 抽检10%确认 |

**fuzzy匹配失败后的人工兜底责任人：项目方（业务人员）**

### 2.3 业务方需确认什么（业务方负责）

| # | 业务方负责事项 | 说明 |
|---|--------------|------|
| 1 | 确认映射结果与业务判断一致 | 抽检匹配结果 |
| 2 | 确认结论卡中黄灯/结项判断是否准确 | B3准确率验证的一部分 |

## 三、映射数据生成规则

### 3.1 若有现成映射JSON（推荐）

**由项目方提供 mapping JSON 文件，格式如下：**
```json
[
  {
    "period": "0414-0418",
    "source_type": "key_alerts",
    "source_project_name": "XX项目",
    "match_type": "exact",
    "ledger_record_id": "ledger-01",
    "ledger_project_name": "XX项目名称"
  }
]
```

**提供现成映射JSON的优势：**
- 跳过系统自动匹配，避免匹配错误
- 准确率最高
- 只需提供一次，后续可复用

### 3.2 若无现成映射JSON（系统自动+人工兜底）

**第一步：系统自动fuzzy匹配**
```python
# 系统自动输出未匹配清单
unmatched = [
  {"period": "0414-0418", "source_project_name": "XX项目", "reason": "fuzzy_no_match"}
]
```

**第二步：项目方人工补充失败项**
项目方根据"XX项目"在台账中的实际名称，补全 ledger_record_id

**第三步：项目方评估fuzzy匹配准确率**
抽检10%的匹配结果，确认无明显错误

## 三、映射数据责任边界

| # | 映射场景 | 责任方 | 说明 |
|---|---------|--------|------|
| 1 | 周报事项 → 台账项目匹配 | **系统自动（推荐）** | 使用现有fuzzy匹配逻辑 |
| 2 | fuzzy匹配失败的项 | **项目方** | 人工补充映射或调整项目名称 |
| 3 | 映射关系提供 | **项目方** | 如有现成的映射文件，提供JSON格式 |
| 4 | 映射质量验证 | **项目方+开发方** | 共同确认匹配准确率 |

---

## 四、真实数据阶段的映射策略

### 4.1 推荐策略：系统自动优先 + 人工兜底

```
Step 1: 系统自动fuzzy匹配
    ↓
Step 2: 输出匹配结果（成功/失败列表）
    ↓
Step 3: 项目方确认失败项
    ↓
Step 4: 人工补充失败项映射 或 调整周报/台账项目名称
    ↓
Step 5: 最终匹配结果进入后续处理
```

### 4.2 替代策略：全人工标注

**适用场景：** 项目名称规律性差，fuzzy匹配准确率低

**工作量：** 需标注全部周报事项（每期60+事项）

**不建议原因：** 工作量大，且系统自动匹配在大多数情况下已足够好

## 五、映射缺失会影响哪些输出

| # | 输出项 | 受映射影响程度 | 说明 |
|---|--------|-------------|------|
| 1 | 事项总台账 | **高** | 台账中找不到对应项目 → 事项游离在外 |
| 2 | 领导摘要 | **中** | 事项与台账关联丢失 → 摘要内容不完整 |
| 3 | 风险清单 | **低** | 风险判断主要依赖周报数据 |
| 4 | 结论卡 | **中** | 结论判断需要台账状态作为依据 |
