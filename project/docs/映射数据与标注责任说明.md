**样本路径：** `samples/samples/linked/weekly_to_ledger_mapping_latest3.json`


| ledger_record_id | 匹配到的台账record_id |
| ledger_project_name | 台账中的项目名称 |

## 二、映射的处理方式：系统自动 vs 人工标注

### 2.1 系统自动映射（已有）

**现有代码已有fuzzy匹配逻辑：**

```python
# sample_loader.py
def find_ledger_record(ledger_index, project_name):
    """在台账中模糊匹配项目名称"""
    name_lower = project_name.lower()
    for rid, rec in ledger_index.items():
        if name_lower in rec['project_name'].lower():
            return rec
    return None
```

**系统自动映射的能力：**
- 按项目名称fuzzy匹配（周报事项 ↔ 台账项目）
- 匹配成功 → 返回台账记录
- 匹配失败 → 记录为"未匹配"
- 不需要人工逐条标注

**系统自动映射的限制：**
- 项目名称必须有规律性（如"XX项目"在周报和台账中基本一致）
- 名称差异过大（如周报写"贷款方案"、台账写"信贷优化方案"）会导致匹配失败
- 无法处理多对一的情况（多个周报事项对应同一个台账项目）

### 2.2 人工标注映射

**需要人工标注的情况：**
1. 系统fuzzy匹配失败的项目
2. 需要精确匹配而非模糊匹配的场景
3. 多对一映射需要明确优先级
4. 特殊缩写或简称映射

**人工标注的工作量：**
- 取决于周报事项数量和台账项目名称的规律性
- 样本数据（3期，60+事项）：约12条映射关系，人工需确认约3-5条
- 真实数据：取决于项目数量

## 三、映射数据责任边界

| # | 映射场景 | 责任方 | 说明 |
|---|---------|--------|------|
| 1 | 周报事项 → 台账项目匹配 | **系统自动（推荐）** | 使用现有fuzzy匹配逻辑 |
| 2 | fuzzy匹配失败的项 | **项目方** | 人工补充映射或调整项目名称 |
| 3 | 映射关系提供 | **项目方** | 如有现成的映射文件，提供JSON格式 |
| 4 | 映射质量验证 | **项目方+开发方** | 共同确认匹配准确率 |

---

## 四、真实数据阶段的映射策略

### 4.1 推荐策略：系统自动优先 + 人工兜底

```
Step 1: 系统自动fuzzy匹配
    ↓
Step 2: 输出匹配结果（成功/失败列表）
    ↓
Step 3: 项目方确认失败项
    ↓
Step 4: 人工补充失败项映射 或 调整周报/台账项目名称
    ↓
Step 5: 最终匹配结果进入后续处理
```

### 4.2 替代策略：全人工标注

**适用场景：** 项目名称规律性差，fuzzy匹配准确率低

**工作量：** 需标注全部周报事项（每期60+事项）

**不建议原因：** 工作量大，且系统自动匹配在大多数情况下已足够好

## 五、映射缺失会影响哪些输出

| # | 输出项 | 受映射影响程度 | 说明 |
|---|--------|-------------|------|
| 1 | 事项总台账 | **高** | 台账中找不到对应项目 → 事项游离在外 |
| 2 | 领导摘要 | **中** | 事项与台账关联丢失 → 摘要内容不完整 |
| 3 | 风险清单 | **低** | 风险判断主要依赖周报数据 |
| 4 | 结论卡 | **中** | 结论判断需要台账状态作为依据 |
