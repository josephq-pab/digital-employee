# 确认反馈合并规则

**版本：** v1.0
**日期：** 2026-04-07
**用途：** 外部反馈回来后，如何一次性并入工程文档，不再反复打架

---

## 一、反馈接收原则

### 1.1 唯一入口

所有外部确认反馈统一由用户接收后录入，Agent不直接接收外部反馈。

### 1.2 反馈分类

| 类型 | 定义 | 处理优先级 |
|------|------|----------|
| 必须确认类 | MGT-01 / PMO-01 / PMO-02 / ADPT-02 / ADPT-03 / PILOT-01 | P0，立即处理 |
| 可调整类 | ADPT-01 / MAP-01~04 | P1，本周内处理 |
| 建议类 | ADPT-04 / MAP-04 | P2，可暂缓 |

### 1.3 反馈接收后的标准动作（5步）

```
Step 1：登记
  → 打开《确认结果回填模板.md》
  → 逐项填写：同意/修改后同意/不同意/暂缓

Step 2：判断
  → 确认类型属于哪类（P0/P1/P2）
  → 判断是否有修改建议

Step 3：优先级排序
  → 多个反馈同时到达时：
     管理层反馈 > PMO反馈 > 样本方反馈

Step 4：更新文档
  → 同意类 → 直接更新对应工程文档
  → 修改后同意类 → 先更新工程文档，再更新门控
  → 不同意类 → 登记并发起升级讨论

Step 5：更新门控
  → 执行《确认结果回填模板》→ 驱动门控卡更新
```

## 二、反馈优先级排序规则

### 2.1 横向优先级

| 优先级 | 反馈来源 | 适用场景 |
|--------|---------|---------|
| P0 | 管理层 | MGT-01（输出格式）|
| P1 | PMO | PMO-01/PMO-02（术语/规则）|
| P2 | PMO | ADPT-01~04 / MAP-01~04（适配性）|
| P3 | 样本方 | 样本质量问题 | 

### 2.2 纵向冲突解决规则

如果管理层与PMO意见冲突：

| 场景 | 解决规则 |
|------|---------|
| 管理层选择模板B，PMO要求模板A | 以管理层意见为准，记录PMO保留意见 |
| 管理层要求修改术语，PMO认为应保持 | 记录双方意见，由用户决定最终版本 |
| PMO与样本方数据不一致 | 以样本方原始数据为准，PMO负责解释差异 |

**核心原则：谁的服务对象级别高，谁的意见优先。**

## 三、不同反馈类型的处理规则

### 3.1 同意类（全部同意）

**处理规则：**
1. 直接采用默认建议
2. 对应工程文档立即冻结该条款
3. 门控卡对应项立即更新为"已确认"
4. 填写《确认结果回填模板》

**需要更新的文档：**
- 《判断规则与状态机说明.md》（如涉及规则）
- 《输入输出数据契约.md》（如涉及字段）
- 《阶段4B进入门控卡.md》（门控更新）

---

### 3.2 修改后同意类

**处理规则：**
1. 记录具体修改内容
2. 判断修改影响范围（规则/字段/展示）
3. 确定修改是否影响已完成的冒烟测试
4. 如影响冒烟测试 → 需重新执行冒烟测试
5. 更新所有受影响文档
6. 门控卡暂不更新，直至修改后的内容被接受

**处理时限：**
- 轻微修改（措辞/格式）：1个工作日
- 中度修改（字段/规则调整）：3个工作日
- 重度修改（规则引擎变更）：5个工作日

---

### 3.3 不同意类

**处理规则：**
1. 记录不同意原因和替代建议
2. 评估不同意内容的合理性
3. 如合理 → 发起与确认方的协商会议
4. 如不合理 → 记录保留意见，维持原方案
5. **不得单方面推翻已冻结的条款**

---

### 3.4 暂缓类

**处理规则：**
1. 登记为"暂缓，确认时间待定"
2. 门控卡对应项标注为"暂缓，不阻断4B"
3. 设定提醒时间（建议1个月后再次询问）
4. 不影响其他确认项的推进

---

## 四、多个文档同步更新规则

当一条反馈影响多个文档时：

| 影响类型 | 影响的文档 | 更新顺序 |
|---------|----------|---------|
| 术语变更 | 术语口径 + 规则说明 + 数据契约 | 术语口径 → 规则说明 → 数据契约 |
| 规则变更 | 规则说明 + engine.py + 冒烟测试 | 规则说明 → engine.py → 冒烟测试 |
| 字段变更 | 数据契约 + INPUT_SCHEMA + 适配层 | 数据契约 → INPUT_SCHEMA → 适配层 |
| 输出格式变更 | 数据契约 + 冒烟测试 + 演示版 | 数据契约 → 冒烟测试 → 演示版 |

**更新原则：先更新源头文档，再更新派生文档。**

## 五、门控联动规则

| 反馈结果 | 门控更新动作 |
|---------|------------|
| MGT-01确认 | G2 → 🟢 |
| PMO-01+PMO-02确认 | G1 → 🟢 |
| ADPT-02+ADPT-03确认 | G1 → 🟢（共同组成G1） |
| PILOT-01确认 | G3 → 🟢 |
| 全部P0类确认完成 | 总体 → 🟢（可进入4B） |
| 暂缓类反馈 | 门控卡标注"暂缓，不阻断"，继续推进 |
| 修改后同意 | 门控卡暂保持🟡，修改完成后再更新 |

## 六、反馈记录留存规则

所有确认反馈必须留存：
- 原始反馈截图/邮件（存于《确认反馈原始记录.md》）
- 填写后的《确认结果回填模板.md》
- 记录文档变更的《变更记录.md》

**留存时间：** 试点结束+1年
