# 确认反馈快捷处理分支

**版本：** v1.0
**日期：** 2026-04-07
**用途：** 收到反馈后，三种情况的最小动作集

---

## 分支概览

| 分支 | 条件 | 核心动作 |
|------|------|---------|
| **分支A：全部通过** | 6项全部"同意" | 宣布→回填→门控→冻结→检查→开工 |
| **分支B：部分修改** | 1-3项"修改后同意" | 记录→更新文档→重新冒烟→继续等待→门控暂🟡 |
| **分支C：关键否决** | 任一"不同意" | 记录→暂停→发起协商→门控维持🔴 |

---

## 分支A：全部通过

**条件：MGT-01 + PMO-01 + PMO-02 + ADPT-02 + ADPT-03 + PILOT-01 全部"同意"**

### 最小动作集（按顺序，30分钟内完成）

```
[1] 用户宣布："全部确认完成，门控可进入🟢"
[2] Agent填《确认结果回填模板》（6项→✅）
[3] Agent更新《阶段4B进入门控卡》（G1/G2/G3→🟢，总体→🟢）
[4] Agent执行《4B开工前最后检查表》（≥28/32通过）
[5] 如通过 → Agent发出"T01开工通知"
[6] 用户确认开工 → Agent开始T01
```

**无任何分支判断，直接线性执行。**

---

## 分支B：部分修改

**条件：1-3项收到"修改后同意"，其余为"同意"**

### 最小动作集

```
[1] 识别"修改后同意"的项（1-3项）
[2] 记录修改内容（填入《确认结果回填模板》备注栏）
[3] 评估修改影响：
     - 如修改影响engine.py → 需重新冒烟测试
     - 如修改仅影响展示 → 更新后继续
[4] 更新受影响的文档（1个工作日内）
[5] 如需重新冒烟 → 执行smoke_tests.json → 确认通过
[6] 门控更新：
     - "修改后同意"项 → 暂维持🟡
     - 全部修改完成后 → 门控升级
[7] 继续等待其他项确认（不影响已确认项的进展）
```

**注意：部分修改不阻断其他已确认项的推进。**

---

## 分支C：关键否决

**条件：任一确认项收到"不同意"**

### 最小动作集

```
[1] 立即识别"不同意"项（哪个包，哪条确认项）
[2] 记录不同意原因（《确认结果回填模板》备注栏）
[3] 门控卡不变（对应G维持🔴）
[4] 判断不同意性质：
     - 轻微（措辞调整）→ 协调修改，1-2天内解决
     - 中度（字段变更）→ 需重新冒烟测试，3-5天
     - 重度（推翻核心规则）→ 发起多方协商会议
[5] 发起协商（用户执行）：
     - 联系确认方，了解具体意见
     - 提出替代方案
[6] 等待协商结果
[7] 协商完成后：
     - 如接受 → 按"分支B修改后同意"处理
     - 如拒绝 → 继续协调或升级
```

**关键原则：任一"不同意"=门控总体不得变为🟢，不得进入4B。**

---

## 三种分支的对比

| 维度 | 分支A（全部通过） | 分支B（部分修改） | 分支C（关键否决） |
|------|-----------------|-----------------|-----------------|
| 门控 | 立即→🟢 | 暂维持🟡 | 维持🔴 |
| 进入4B | 立即可启动 | 等修改完成 | 暂停 |
| 等待时间 | 无 | 1-5天 | 不确定 |
| 需重新冒烟 | 否 | 视情况 | 可能 |
| 需协商 | 否 | 否 | 是 |

---

## 反馈结果速查

| 收到结果 | 执行分支 | 门控 | 进入4B |
|---------|---------|------|--------|
| 全部"同意" | A | 🟢 | 可立即 |
| 1-3项"修改后同意" | B | 🟡 | 等修改完成后 |
| 任一"不同意" | C | 🔴 | 暂停 |

---

## 快速决策表（收到反馈时）

```
问：这个反馈是"同意"吗？
  → 是：执行分支A
  → 否：是"修改后同意"吗？
       → 是：执行分支B
       → 否：是"不同意"吗？
            → 是：执行分支C
            → 否：是"暂缓"吗？
                 → 记录，暂不触发门控变化
```
