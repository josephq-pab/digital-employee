# 确认反馈处理SOP

**版本：** v1.0
**用途：** 当确认包收到反馈后，如何处理、如何更新文档、如何更新门控卡

---

## 一、反馈处理总流程

```
收到确认反馈
    │
    ▼
判断反馈类型（见第二章）
    │
    ├── 直接通过 → 进入处理流程A
    ├── 小修改后通过 → 进入处理流程B
    ├── 需重提 → 进入处理流程C
    └── 暂缓确认 → 进入处理流程D
    │
    ▼
执行对应处理流程
    │
    ▼
更新《确认包签收与反馈登记表》
    │
    ▼
更新《决策记录与待确认事项.md》
    │
    ▼
更新《阶段4B进入门控卡》
    │
    ▼
（如全部Q01-Q05均通过）→ 检查是否可进入4B
```

---

## 二、四种反馈类型及处理

### 类型A：直接通过

**判断标准：** 接收人签字确认，无修改意见

**处理流程：**
```
1. 收取签字确认件（扫描/原件）
2. 在《确认包签收与反馈登记表》中登记：反馈结果=通过
3. 在《决策记录与待确认事项.md》中：
   - 将对应Q状态更新为"已确认"
   - 记录确认日期和确认依据
4. 更新《阶段4B进入门控卡》：对应条件→🟢
5. 通知相关方：确认已完成
6. 归档签字确认件（扫描存档）
```

---

### 类型B：小修改后通过

**判断标准：** 接收人确认，但有少量文字/格式修改意见，不影响核心逻辑

**处理流程：**
```
1. 记录修改意见（逐条记录，不得遗漏）
2. 评估修改影响范围：
   - 是否影响核心规则 → 影响大，走变更流程
   - 是否仅影响措辞 → 影响小，直接修改
3. 执行修改：
   - 更新对应文档
   - 记录修改内容和修改原因
4. 在《确认包签收与反馈登记表》中登记：反馈结果=修改后通过
5. 更新《决策记录与待确认事项.md》：Q状态="已确认（含修改）"
6. 更新《阶段4B进入门控卡》：对应条件→🟢
7. 通知接收人：修改已完成，请确认
8. 如接收人再次确认 → 完成；如不接受 → 进入类型C
```

---

### 类型C：需重提

**判断标准：** 接收人对核心内容有较大异议，需重新提出或修改后重新确认

**处理流程：**
```
1. 记录异议内容（详细记录，不得省略）
2. 评估异议：
   - 是否属于本阶段可解决范围
   - 是否需要推翻已确认事项
3. 处理方式：
   a. 如属合理异议 → 修正后重新发出确认包
      - 更新确认包
      - 重新发出
      - 重新登记发送日期
   b. 如属不合理异议 → 记录，向用户汇报，由用户决定
4. 在《确认包签收与反馈登记表》中登记：反馈结果=退回重提
5. 更新《阶段4B进入门控卡》：对应条件→🔴
6. 向用户报告异议内容和处理方案
```

---

### 类型D：暂缓确认

**判断标准：** 接收人确认收到，但当前无法给出正式反馈

**处理流程：**
```
1. 记录暂缓原因和预计反馈时间
2. 在《确认包签收与反馈登记表》中登记：反馈结果=暂缓，确认预计反馈日期
3. 设定跟进提醒（在预计反馈日期前1个工作日提醒）
4. 在预计反馈日期前主动联系接收人
5. 后续收到反馈 → 重新进入本流程处理
```

## 四、反馈处理与门控卡联动

| 反馈类型 | 门控卡动作 |
|---------|---------|
| A：直接通过 | 对应G→🟢 |
| B：小修改后通过 | 对应G→🟢（附修改记录） |
| C：需重提 | 对应G→🔴，重启发出流程 |
| D：暂缓确认 | G状态不变，超期后→🔴 |

## 五、全部确认通过后的标准动作

当Q01、Q03、Q04全部确认通过，S1、S2样本到位：

```
1. 更新《决策记录与待确认事项.md》：
   - Q01状态→已确认
   - Q03状态→已确认
   - Q04状态→已确认
   - Q02→已确认（S1到位）
   - Q05→已确认（S2到位）

2. 更新《阶段推进总表》：
   - 阶段2关闭条件满足
   - 阶段3退出条件满足
   - 阶段4进入条件满足

3. 更新《阶段4B进入门控卡》：
   - G1→🟢
   - G2→🟢
   - G3→🟢
   - G4→🟢
   - G5→🟢
   - G6→🟢
   - G7→🟢

4. 输出通知：
   - "阶段4B进入条件全部满足，可以开始实现"
   - 通知所有相关方
```
