# 试点运行机制与人工确认SOP

**版本：** v1.0
**适用阶段：** 阶段3试点运行期（阶段4实现后进入试点，阶段5持续）
**目的：** 在不进入开发的前提下，定义试点期间的运行机制，供后续实现时参考

---

## 一、角色定义

| 角色 | 缩写 | 职责 |
|------|------|------|
| 填报人 | FU（Filing User） | 项目责任人，负责定期填报周报/月报/台账 |
| PMO协调员 | PC（PMO Coordinator） | 负责收样本、催填报、记录问题 |
| 审阅人 | RV（Reviewer） | 管理层或指定负责人，负责审阅系统输出 |
| Agent | AG（Agent） | 执行系统处理（阶段4实现后由系统自动执行） |

---

## 二、周度运行SOP

### 2.1 周期定义

- 启动时间：每周一（或约定日）早9:00前
- 截止时间：每周一晚18:00前完成数据提交

### 2.2 周度流程

```
【周一 09:00前】PC 发出填报提醒
    │
    ▼
【周一 18:00前】FU 提交周报（JSON格式或表单）
    │
    ▼
【周一 18:00后】PC 验收样本
    │  ├─ 通过 → 提交给 AG
    │  └─ 不通过 → 退回 FU 补充（注明缺口）
    │
    ▼
【周一当晚】AG 执行处理（系统自动）
    │  输入解析 → 事实提取 → 规则判断 → 摘要生成
    │
    ▼
【周二早9:00前】AG 输出领导摘要JSON + 文本
    │
    ▼
【周二 12:00前】RV 审阅摘要
    │  ├─ 接受 → 存档，流转使用
    │  ├─ 有异议 → 手动标注，存档+反馈给FU
    │  └─ 重大异议 → 触发问题登记，进入纠偏流程
    │
    ▼
【周二 17:00前】RV 审阅完成
    │
    ▼
【周二 17:00后】PC 整理周度记录，关闭本周闭环
```

### 2.3 周度交付物

| 交付物 | 格式 | 接收方 | 存档位置 |
|--------|------|--------|---------|
| 领导摘要 | JSON + 文本 | RV → 管理层 | PMO存档 |
| 问题记录 | JSON | PC | data/issues/ |
| 周度统计 | 文本摘要 | PC | data/weekly_summaries/ |

## 三、月度运行SOP

### 3.1 周期定义

- 汇总周期：每月1日 ～ 月末
- 月度报告生成时间：次月第一个工作日

### 3.2 月度流程

```
【每月末】PC 汇总本月全部周报台账
    │
    ▼
【次月第一个工作日】AG 生成月度汇总
    │  - 汇总本月风险等级分布
    │  - 汇总升级建议采纳情况
    │  - 汇总无明确进展项目
    │
    ▼
【次月第二个工作日】RV 审阅月度汇总
    │
    ▼
【月度会议（如有）】RV 汇报整体态势
```

### 3.3 月度交付物

| 交付物 | 格式 | 接收方 |
|--------|------|--------|
| 月度风险清单 | JSON列表 | 管理层 |
| 月度状态概览 | 文本 | 管理层 |
| 试点运行月报 | 文本 | PMO |

## 四、人工审阅节点（不可跳过）

### 4.1 审阅内容清单

| 审阅项 | 审阅问题 | 通过标准 |
|--------|---------|---------|
| 事实提取 | 系统提取的事实是否准确反映原始输入？ | 无误解或遗漏关键信息 |
| 风险等级 | 风险等级（低/中/高）是否合理？ | 人工可判断，无明显错误 |
| 升级建议 | escalation_recommended是否合理？ | 人工可判断 |
| 遗漏判断 | 是否有明显风险被系统漏判？ | 人工根据经验判断 |

### 4.2 审阅结果分类

| 审阅结果 | 含义 | 后续动作 |
|---------|------|---------|
| **采纳** | 系统输出准确，直接使用 | 存档 |
| **修正后采纳** | 大方向对，有局部错误，人工修正后使用 | 存档+记录修正内容 |
| **拒绝重跑** | 系统逻辑错误，需修正后重跑 | 记录错误，进入问题登记 |
| **升级决策** | 系统建议升级，但人工判断不升级 | 记录人工决策理由 |

## 五、问题登记与纠偏SOP

### 5.1 必须登记的问题

| 问题类型 | 定义 | 登记时限 |
|---------|------|---------|
| 规则误判 | 规则判断结果与人工判断明显不符 | 24小时内 |
| 规则缺失 | 现有规则无法覆盖某类场景 | 24小时内 |
| 字段歧义 | 输入字段含义模糊导致处理异常 | 48小时内 |
| 样本异常 | 样本数据质量影响判断 | 24小时内 |
| 流程异常 | 填报人未按时提交等流程问题 | 次日内 |

### 5.2 问题记录格式

```json
{
  "issue_id": "I-2026-04-001",
  "date_detected": "2026-04-07",
  "detected_by": "[角色缩写]",
  "type": "rule_misjudgment | rule_gap | field_ambiguity | sample_quality | process",
  "description": "问题描述",
  "input_affected": "受影响的具体输入内容",
  "system_output": "系统当时的输出",
  "expected_output": "人工认为正确的输出",
  "resolution": "resolved | in_progress | escalated | wont_fix",
  "resolution_notes": "处理说明",
  "resolved_by": "[处理人]",
  "resolved_date": "ISO8601"
}
```

### 5.3 问题处理规则

| 问题类型 | 可自行迭代 | 需PMO确认 | 需重新验证 |
|---------|----------|-----------|-----------|
| 规则误判 | ⚠️ 需记录后修正 | ✅ 重大误判需确认 | ✅ 修正后需重跑 |
| 规则缺失 | 记录，延后处理 | 需确认是否本期纳入 | 确认后实现 |
| 字段歧义 | 不解决 | 需明确字段口径 | 不涉及 |
| 样本异常 | 降级使用或退回 | 需FU补充 | 不涉及 |

## 六、试点期间升级决策规则

### 6.1 系统升级建议 vs 人工最终决定

```
系统 escaltion_recommended = true
    │
    ▼
人工判断：
  ├─ 采纳系统建议 → 执行升级，向领导汇报
  ├─ 否决系统建议 → 记录理由，继续监控
  └─ 无法判断 → 升级，争取更多信息
```

### 6.2 升级后的问题追踪

- 升级后由RV记录：升级原因、系统判断是否正确、后续结果
- 连续2次升级均被人工否决 → 触发规则复盘

## 七、试点结束条件

| 条件 | 责任方 | 达标标准 |
|------|--------|---------|
| 连续4周运行无重大流程异常 | PC | 每周均完成闭环 |
| PMO书面反馈"可接受继续" | PMO | 反馈文件存档 |
| 规则误判率<20% | AG | 误判次数/总判断次数<20% |
| 管理层连续2个月审阅 | 管理层 | 有审阅记录 |

## 八、试点运行检查清单

**每次周度闭环前，PC必须确认：**

- [ ] 本周样本已提交
- [ ] 样本已通过验收
- [ ] 系统输出已生成
- [ ] RV已完成审阅
- [ ] 审阅结果已记录
- [ ] 本周问题已登记（如有）
- [ ] 周度交付物已存档
