## 三、适配层处理能力检查

### 3.1 alert_lines 解析 ✅

`adapt_alert_line_to_key_alert()` 能正确：
- 识别"黄灯继续提示"/"黄灯提示" → alert_level=yellow
- 提取项目名称（第一个逗号/顿号前的内容）
- 保留完整 raw 文本

### 3.2 request_lines 解析 ⚠️

`adapt_request_lines_to_closure_requests()` 存在以下问题：

**W3-P1（适配层bug）**：
- request_lines 中"2、搭建战略客户样板和北京分行战客专班建设，战客部本周已邮件汇报...申请结项"
  → 提取出的 project_name 为"搭建战略客户样板和北京分行战客专班建设"（整个句子）
  → 实际应为两条独立事项：①"搭建战略客户样板"（申请结项）；②"北京分行战客专班建设"（黄灯，逾期）
  → 导致结项申请中混入了非结项内容，且"北京分行战客专班建设"在黄灯列表中出现时项目名被放大

**W3-P2（适配层bug）**：
- 1117-1121 的"数字经营"延期申请 → 正则 `^\d+[.、](.+?)[，,]` 未匹配到（因项目名以引号开头，无编号前缀）
  → "数字经营"延期申请未被捕获为独立事项
  → 仅作为 management_summary.delayed_items 落入风险信号，未单独识别为"延期申请"类型

### 3.3 台账快照匹配 ✅

68 条台账主记录（4 period 合计），按 `canonical_key` 模糊匹配：
- "北京分行战客专班建设" → 已匹配（ledger_record_id: 1103-1107_北京分行战客专班建设）
- 其他行业类事项 → 未映射（无精确匹配）
