# 4B触发前联动规则

**版本：** v1.0
**日期：** 2026-04-07
**用途：** 6项最小确认路径全部同意后，第一时间触发4B准备动作

---

## 一、触发条件

**触发条件：MGT-01 + PMO-01 + PMO-02 + ADPT-02 + ADPT-03 + PILOT-01 全部"同意"或"修改后同意"**

```
G1（MGT-01） ✅
G2（PMO-01+02+ADPT-02+ADPT-03） ✅
G3（PILOT-01） ✅
         ↓
    总体 → 🟢
         ↓
    触发4B准备
```

---

## 二、触发后动作序列（按顺序执行）

### 第1步：宣布（用户执行）

**动作：**
- 用户宣布："6项确认全部完成，门控总体🟢，可进入4B"
- 发出"确认链已闭合，可进入4B"通知

**执行人：** 用户（Agent只能建议，不能替代宣布）

---

### 第2步：回填（Agent执行）

**动作：**
1. 打开《确认结果回填模板》
2. 将6项全部标记为"同意"/"修改后同意"
3. 填写最终门控汇总栏
4. 状态：全部6项 ✅

**执行人：** Agent

---

### 第3步：门控更新（Agent执行）

**动作：**
1. 打开《阶段4B进入门控卡.md》
2. 更新G1 → 🟢（MGT-01✅）
3. 更新G2 → 🟢（PMO-01/02/ADPT-02/ADPT-03 ✅）
4. 更新G3 → 🟢（PILOT-01 ✅）
5. G4/G5 → 🟢（Q01确认后自动升级）
6. 总体 → 🟢
7. 记录"最后更新时间"=今天

**执行人：** Agent

---

### 第4步：冻结文档（Agent执行）

**动作：**
按《确认链到实现链的切换说明.md》中的9份冻结文档清单：
1. 确认INPUT_SCHEMA.md已冻结
2. 确认OUTPUT_SCHEMA.md已冻结
3. 确认engine.py未修改（冒烟测试通过版本）
4. 记录冻结状态

**执行人：** Agent

---

### 第5步：执行开工前最后检查（Agent执行）

**动作：**
执行《4B开工前最后检查表.md》中的32项检查
- 目标：≥28/32项通过
- 如通过 → 进入第6步
- 如不通过 → 修复未通过项 → 重新检查 → 第6步

**执行人：** Agent

---

### 第6步：发出开工通知（用户执行）

**动作：**
- 用户/Agent发出"T01样本装载开工"通知
- Agent正式进入T01（样本装载与适配层）

**执行人：** Agent发出通知，用户确认

---

## 三、门控从🟡到🟢的更新顺序

```
G2（MGT-01） → 更新 → G2=🟢
     ↓
G1（PMO-01+02+ADPT-02+ADPT-03） → 更新 → G1=🟢
     ↓
G3（PILOT-01） → 更新 → G3=🟢
     ↓
G4/G5（Q01确认后自动升级） → 更新 → G4=🟢, G5=🟢
     ↓
总体 → 🟢
```

|------|------|

## 七、6项中部分项为"修改后同意"时的处理

**"修改后同意"项数 ≤ 2：**
- 先完成文档更新
- 再执行门控更新
- 整体进入4B时间延迟1-3天

**"修改后同意"项数 ≥ 3：**
- 先完成文档更新
- 重新执行冒烟测试验证
- 确认测试通过后再进4B
- 整体进入4B时间延迟3-5天
