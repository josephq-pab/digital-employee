# 冒烟测试计划与通过标准

**版本：** v1.0
**目的：** 定义冒烟测试的目标、能力覆盖、通过标准、归因方法和记录规范

---

## 一、冒烟测试目标

> 用最少的测试用例验证核心规则在已知场景下按预期工作，确认"规则推理→判断→输出"链路基本可用。

**目标边界：**
- ✅ 验证核心规则逻辑正确
- ✅ 验证输入解析基本可用
- ✅ 验证输出JSON结构完整
- ❌ 不验证真实样本处理能力
- ❌ 不验证长期稳定性
- ❌ 不验证PMO满意度

---

## 二、冒烟测试能力矩阵

### 2.1 最小能力集（必须全部通过）

| # | 能力 | 输入 | 预期输出 | 验证什么 |
|---|------|------|---------|---------|
| A1 | 样本可读取 | JSON文件 | 无解析报错 | InputParser能正确加载JSON |
| A2 | 字段可解析 | 格式A JSON | InternalRecord生成 | 字段映射正确 |
| A3 | 事项可抽取 | InternalRecord | Facts列表 | FactExtractor正确提炼事实 |
| A4 | 基本规则可判断 | Facts + InternalRecord | JudgementResult | RuleEngine返回结果 |
| A5 | 领导摘要可生成 | JudgementResult | JSON含所有必须字段 | SummaryGenerator输出完整 |
| A6 | 台账可生成（内部） | InternalRecord | items列表 | 台账生成逻辑存在 |
| A7 | 风险清单可生成（内部） | InternalRecord | risks列表 | 风险清单生成逻辑存在 |

### 2.2 扩展能力集（目标：20条用例，覆盖更多场景）

| 场景 | 覆盖内容 |
|------|---------|
| 正常类 | TC01（无风险+正常进度） |
| 滞后类 | TC02（滞后+风险）、TC04（严重滞后）、TC08（停滞）、TC09（未完成） |
| 升级类 | TC03（escalation=true） |
| 边界类 | TC06（空risk_points）、TC07（escalation=false显式）、TC14（模糊进度） |
| 无进展类 | TC19（空进展）、TC20（无进展词） |
| 格式类 | TC16（格式B台账）、TC17（台账+风险）、TC18（台账+升级） |
| 长度类 | TC15（长文本） |

## 三、通过标准

### 3.1 冒烟通过门槛

| 条件 | 标准 |
|------|------|
| 最小能力集（A1-A7） | **必须100%通过（7/7）** |
| 扩展能力集（目标20条） | **建议通过率≥80%（16/20）** |

### 3.2 每条测试通过判定

| 检查项 | 标准 |
|--------|------|
| risk_level | 与预期完全一致 |
| escalation_recommended | 与预期完全一致 |
| overall | 与预期完全一致 |
| facts_extracted | 非空列表 |
| rules_applied | 非空列表 |

### 3.3 失败归因

| 失败模式 | 归因结果 | 后续动作 |
|---------|---------|---------|
| TC01-TC05（核心规则）失败 | 核心逻辑错误 | 立即修复，不得继续 |
| TC06-TC20失败 | 边界处理缺陷 | 记录，在阶段5前修复 |
| facts_extracted为空 | FactExtractor错误 | 立即修复 |
| rules_applied为空 | RuleEngine错误 | 立即修复 |

## 四、测试结果记录

### 4.1 记录要求

每次 run_smoke.sh 执行后，必须生成：

```json
{
  "run_id": "smoke-run-2026-04-07-001",
  "executed_at": "ISO8601",
  "total_cases": 20,
  "passed": 18,
  "failed": 2,
  "pass_rate": "90%",
  "results": [
    {
      "case_id": "TC01",
      "passed": true,
      "risk_level_actual": "低",
      "risk_level_expected": "低",
      "escalation_actual": false,
      "escalation_expected": false,
      "overall_actual": "正常",
      "overall_expected": "正常",
      "failure_reason": null
    }
  ]
}
```

### 4.2 存档位置

- 测试结果存档：`eval/smoke_test_results/{date}.json`
- 最新结果：`eval/smoke_test_results/latest.json`

## 五、测试是否需要人工核验

**最小能力集（A1-A7）：不需要人工核验，自动化判定。**

**扩展能力集（TC06-TC20）：建议在阶段5前完成人工抽检（至少抽检5条），确认系统判定与人工判断一致。**

## 六、测试环境要求

| 环境项 | 要求 |
|--------|------|
| Python版本 | ≥3.8 |
| 外部依赖 | 零（原型阶段不使用任何第三方库） |

| 执行方式 | bash scripts/run_smoke.sh |
| 数据准备 | 20条测试用例JSON文件 |

---

## 七、冒烟测试执行流程

```
scripts/run_smoke.sh 执行
    │
    ▼
加载 test_cases/smoke_tests.json（A1-A7 + TC01-TC05）
    │
    ▼
逐条执行，记录实际输出 vs 预期输出
    │
    ▼
生成 eval/smoke_test_results/{date}.json
    │
    ▼
判断：
  ├─ A1-A7 全部通过 → 冒烟通过 ✅
  └─ 任意A1-A7失败 → 冒烟不通过，停止继续 🔴
```

---

## 八、冒烟测试与验收的关系

| 结论 | 含义 |
|------|------|
| 冒烟测试100%通过 | 核心逻辑闭环可验证，后续可进入演示版 |
| 冒烟测试失败 | 核心逻辑有错误，阶段4未完成，不得进入阶段5 |
| 扩展用例<80%通过 | 边界处理有缺陷，记录并在阶段5前修复，不阻断阶段5进入 |
