## 五、试点运行机制结论

### 5.1 角色体系

| 角色 | 缩写 | 核心职责 |
|------|------|---------|
| 填报人 | FU | 定期提交周报/月报/台账 |
| PMO协调员 | PC | 收样本、催填报、记录问题 |
| 审阅人 | RV | 审阅系统输出，决定采纳/修正 |
| Agent | AG | 执行处理（阶段4后系统自动） |

### 5.2 周度闭环

```
FU提交周报 → PC验收 → AG处理 → 领导摘要输出 → RV审阅 → 本周闭环
（周一18:00前）  （当日）  （10秒内）  （次日12:00前）  （当日17:00前）
```

### 5.3 人工审阅四种结果

采纳 / 修正后采纳 / 拒绝重跑（触发问题登记）/ 升级决策

## 六、冒烟测试基线结论

### 6.1 最小能力集（A1-A7）

样本可读取 → 字段可解析 → 事项可抽取 → 规则可判断 → 领导摘要生成 → 台账生成（内部） → 风险清单生成（内部）

### 6.2 通过门槛

| 能力集 | 通过标准 |
|--------|---------|
| A1-A7（最小能力，7条） | **必须100%通过**（0条失败可接受） |
| TC06-TC20（扩展能力，15条） | 建议≥80%通过（12/15） |

### 6.3 关键原则

- 冒烟通过≠原型完成（还需演示可运行）
- 扩展用例失败不阻断阶段5进入，但须记录并于试点前修复
- 失败归因：TC01-TC05失败→立即修复；TC06-TC20失败→记录修复

## 七、演示版与MVP边界结论

### 7.1 核心区分

| 概念 | 核心差异 | 验证方式 |
|------|---------|---------|
| 演示版 | 能讲能看 | PMO/管理层书面反馈 |
| MVP | 能用可靠 | PMO持续使用4周 + 书面接受 |

### 7.2 演示必须讲清的6条边界

1. 人工审阅不可跳过
2. 判断依据可查（reasoning字段）
3. 系统不修正输入
4. 规则有限（5条核心）
5. 升级是建议，最终由人工决定
6. 系统不替代人决策

### 7.3 MVP成功标准

连续4周运行 + PMO书面"接受"或"可接受，需改进" + 规则误判率<20% + 管理层连续2个月审阅
