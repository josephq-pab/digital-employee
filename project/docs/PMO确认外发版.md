## 二、必须确认的事项（2项）

### 必须确认(1)：8个业务术语定义

以下术语定义将用于风险判断规则，请确认是否与PMO日常口径一致：

| # | 术语 | 当前定义 | 是否可接受 |
|---|------|---------|-----------|
| 1 | 进度滞后 | 承诺交付时间已过但未完成 |  |
| 2 | 严重滞后 | 承诺交付时间已过且当前无进展或停滞 |  |
| 3 | 风险自述 | 责任人主动提及风险/阻碍/问题 |  |
| 4 | 升级请求 | 责任人明确提出需要领导介入或资源支持 |  |
| 5 | 正常进展 | 有进展描述且无负面关键词 |  |
| 6 | 低风险 | 无滞后、无风险自述、无升级请求 |  |
| 7 | 中风险 | 有进度描述异常或轻度风险自述 |  |
| 8 | 高风险 | 严重滞后、明确升级请求或多个风险叠加 |  |

**如有不一致的术语，请注明您的定义。**

---

### 必须确认(2)：5条核心判断规则

以下规则将直接决定系统对每个项目的状态判断，请确认是否合理：

| # | 规则名称 | 当前逻辑 | 是否可接受 |
|---|---------|---------|-----------|
| R1 | 规则_严重滞后 | 进展中出现"严重滞后"等关键词 → 高风险+升级 |  |
| R2 | 规则_请求介入 | 进展中出现"请求领导介入"/"需要支持"等 → 高风险+升级 |  |
| R3 | 规则_进度滞后 | 进展中出现"滞后"/"延期"/"延迟"等 → 中风险 |  |
| R4 | 规则_风险自述 | 进展中出现"风险"/"问题"/"阻碍"等 → 中风险 |  |
| R5 | 规则_正常进展 | 进展正常且无负面关键词 → 低风险 |  |

**注意：** 规则R1优先于R3（即"严重滞后"会直接触发升级，不会被"滞后"规则覆盖）。

## 三、可在试点中调整的事项（4项）

以下事项如无异议，请回复"同意"；如有调整意见，可在试点第一期后反馈。

### 可调整(1)：风险点提取逻辑

当前方案：从周报 key_alerts 字段 + management_summary 中含风险词的句子提取 risk_points。
- **是否接受？** 如接受请回复"同意"。

### 可调整(2)：escalation_requested 字段缺失的替代方案

当前方案：周报/台账均无 escalation_requested 字段，系统以"请求介入"等关键词自动判断升级需求。
- **是否接受？** 如接受请回复"同意"。

### 可调整(3)：周报无 owner 字段的替代方案

当前方案：周报本身无 owner 字段，以台账 owner_department 作为责任人字段。
- **是否接受？** 如接受请回复"同意"。

### 可调整(4)：行业维度 vs 项目维度的结构差异说明

当前情况：周报按行业维度组织（如"汽车行业黄灯"），台账按项目维度组织，两者设计维度不同，属正常差异。
- **是否接受这是可接受的结构性差异？** 如接受请回复"同意"。
