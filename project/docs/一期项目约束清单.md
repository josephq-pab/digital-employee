# 一期项目约束清单

## 一、一期要解决的核心问题

**必须解决：**
- PMO团队对重点项目进展的被动跟踪问题（依赖人工催报、汇总，无法结构化沉淀）
- 管理层获取项目风险摘要时缺乏标准化口径
- 重点事项的执行状态无法快速形成"可阅、可复核、可追踪"的摘要

## 二、一期明确不做什么

**不得做：**
- 不得接入 OA 系统或项目管理平台（一期仅支持人工填报：周报/月报/项目台账，不做系统自动采集）
- 不得替代人工做最终决策，所有关键判断须有人工确认环节
- 不得实现自动催报、自动通知或自动升级
- 不得支持多项目自动排名或自动打分
- 不得生成图表、Dashboard 或前端界面（一期仅输出结构化文本）
- 不得实现跨系统数据拉取或自动汇聚

## 三、成功的定义

**必须同时满足：**
1. 给定一份结构化项目周报/台账，系统能输出符合《一期术语与判断口径》的标准化摘要
2. 该摘要中每一项判断都有明确的口径依据，人工可复核
3. 冒烟测试 100% 通过（5条核心规则）
4. 演示时，给定一份真实样例，可在 30 秒内输出可读摘要
5. 一期不追求全自动：系统输出后，必须有人工审阅环节

## 四、必须人工确认的事项

**以下判断不得由系统自动闭环：**
- 事项是否真正完成（完成标准可能因项目而异）
- 延期是否可接受（需要业务判断，非纯规则）
- 是否真正需要领导介入（最终决策权在人工）
- 风险等级是否需要升级（系统给出建议，人工确认）

**以下判断可以系统自动完成：**
- 进度是否滞后（基于关键词规则）
- 是否存在风险自述（文本中存在风险关键词）
- 是否请求介入（字段显式声明）
- 事项是否无明确进展（基于进度描述关键词）

## 五、进入开发阶段前必须满足的条件

**以下条件不满足时，项目不得进入阶段4（原型实现）：**
- [ ] 术语口径文件已由业务方（PMO负责人）书面确认
- [ ] 至少3份真实样例数据已到位，且已验证口径可落地
- [ ] 输入规范（最少字段集）已确认
- [ ] 输出格式已获得管理层初步认可
- [ ] 冒烟测试用例已全部定义完毕
- [ ] "人工确认"环节的位置已明确（输出后审阅，而非中途干预）
