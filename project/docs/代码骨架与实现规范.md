## 二、模块边界规范

### 模块边界原则

| 原则 | 说明 |
|------|------|
| 单一职责 | 每个模块只做一件事 |
| 输入输出显式 | 每个函数有明确的输入类型和输出类型 |
| 不跨层调用 | 只能上层调用下层，不得反向 |
| 配置外置 | 关键词/阈值不硬编码，在patterns.py中配置 |

### 模块调用关系

```
main.py
  └─ InputParser.parse()           → InternalRecord
      └─ FactExtractor.extract()    → Facts
          └─ RuleEngine.judge()     → JudgementResult
              ├─ SummaryGenerator.generate()  → Summary（领导摘要）
              ├─ LedgerGenerator.generate()  → Ledger（事项台账）
              └─ RiskListGenerator.generate() → RiskList（风险清单）
```

## 三、日志规范

### 日志级别

| 级别 | 使用场景 |
|------|---------|
| DEBUG | 调试信息（字段值、中间变量） |
| INFO | 正常流程节点（模块进入/退出） |
| WARNING | 可恢复异常（字段截断、降级处理） |
| ERROR | 不可恢复异常（解析失败、规则错误） |

### 日志格式

```python
import logging

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(name)s: %(message)s",
    datefmt="%Y-%m-%d %H:%M:%S"
)
```

### 禁止日志行为

- ❌ 禁止打印完整JSON（可用字段名+长度替代）
- ❌ 禁止打印敏感信息（如脱敏前的原始数据）
- ❌ 禁止在日志中记录密码或密钥

## 四、错误处理规范

### 错误分类

| 错误类型 | 处理方式 | 示例 |
|---------|---------|------|
| InputError | 抛出异常，打印字段缺失提示 | 必填字段为空 |
| ParseError | 抛出异常，打印JSON解析错误 | JSON格式错误 |
| RuleError | 记录ERROR，继续处理下一个规则 | 规则匹配异常 |

### 错误响应原则

- 输入错误 → 明确告知哪个字段有问题
- 解析错误 → 打印原始错误信息
- 规则错误 → 记录并跳过，继续执行

## 五、配置管理规范

## 六、命名规范

### Python文件

- 全小写，下划线分隔：`input_parser.py`
- 禁止：`-`、大写、驼峰

### 类名

- 驼峰：`InputParser`、`FactExtractor`
- 禁止：下划线

### 函数名

- 下划线分隔：`parse_json`、`extract_facts`
- 禁止：驼峰

### 常量

- 全大写，下划线分隔：`MAX_PROGRESS_LEN = 200`

### 变量

- 下划线分隔：小写
- 局部变量：`internal_record`、`facts_list`
- 实例变量：`self.record`

## 七、样本路径规范

所有样本路径使用**相对路径**，不得硬编码绝对路径：

```python
import os

# ✅ 正确：从项目根目录计算
BASE_DIR = os.path.dirname(os.path.dirname(os.path.abspath(__file__)))
SAMPLES_DIR = os.path.join(BASE_DIR, "data", "samples")

# ❌ 错误：绝对路径
# SAMPLES_DIR = "/home/admin/project/data/samples"
```

## 九、最小可实现原则

**核心原则：先跑通，再优化。**

| 阶段 | 原则 |
|------|------|
| 第一层（跑通） | InputParser → RuleEngine → SummaryGenerator，JSON进JSON出 |
| 第二层（补全） | 加入FactExtractor，增加台账/风险清单 |
| 第三层（健壮） | 加入错误处理、日志、边界测试 |
| 第四层（优化） | 不在一期做 |

**在阶段4，不要试图做：**
- 优化代码结构
- 重构已有实现
- 添加额外功能
- 写注释文档
