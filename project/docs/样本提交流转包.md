# 样本提交流转包

**版本：** v1.0
**发出方：** 数字员工项目组
**接收方：** 试点项目责任人（FU）/ PMO协调员（PC）
**用途：** 告诉责任人要交什么、怎么脱敏、怎么命名、交到哪里、验收标准是什么

---

## 一、样本流转总览

```
责任人准备 → 脱敏处理 → 提交给PC → PC验收 → 存档到data/samples/
```

**最短启动路径（S1）：1份样本，1个工作日完成**
**阶段4进入路径（S2）：3份样本，3-5个工作日**

---

## 二、分级样本要求

### S1：最低启动样本（阶段1关闭用，≥1份）

| 项 | 要求 |
|----|------|
| 数量 | ≥1份 |
| 格式 | 周报/月报（格式A）或台账（格式B）均可 |
| 内容 | 必填字段完整 |
| 时间 | 最近一个完整周期 |
| 脱敏 | 必须（见第四章） |
| 用途 | 验证规则逻辑是否与业务匹配 |

### S2：阶段4进入样本（≥3份）

| 项 | 要求 |
|----|------|
| 数量 | ≥3份（可来自同一项目不同周期，或不同项目） |
| 格式 | 周报/月报/台账均可，混合亦可 |
| 内容 | 必填字段完整，建议有risk_points字段 |
| 时间 | 最近1-3个月 |
| 脱敏 | 必须（见第四章） |
| 用途 | 阶段4开发验证、冒烟测试 |

---

## 三、样本必须包含的内容

### 格式A（周报/月报）必填字段

```json
{
  "project_name": "（脱敏后）",
  "owner": "（脱敏后）",
  "report_type": "weekly 或 monthly",
  "promised_items": "本周期承诺完成的工作（≥20字）",
  "current_progress": "本周期实际进展（≥20字）",
  "risk_points": "如有风险请填写，如无则空字符串",
  "escalation_requested": false
}
```

### 格式B（项目台账）必填字段

```json
{
  "project_name": "（脱敏后）",
  "owner": "（脱敏后）",
  "items": [
    {
      "item_id": "事项唯一ID",
      "promised": "承诺里程碑",
      "current_progress": "当前进展",
      "status": "in_progress / completed / blocked"
    }
  ],
  "overall_risk_points": "如有风险请填写，如无则空字符串",
  "escalation_requested": false
}
```

---

## 四、脱敏要求

### 必须脱敏（责任人/PMO执行）

| 字段 | 脱敏方式 | 示例 |
|------|---------|------|
| project_name | 替换为"XX项目N" | "智慧园区一期" → "试点项目A" |
| owner | 替换为常见姓名 | "张明" → "张三" |
| 绝对日期 | 替换为相对日期或留空 | "2026-06-30" → "6月30日" |
| 公司/品牌名 | 替换为"某公司" | "XX科技" → "某公司" |

### 不得脱敏（保持逻辑可验证）

| 字段 | 原因 |
|------|------|
| promised_items描述结构 | 含判断关键词 |
| current_progress描述结构 | 含滞后/完成关键词 |
| risk_points描述结构 | 含风险关键词 |
| 事项数量和粒度 | 影响规则判断 |
| 文字长度比例 | 影响无明确进展判断 |

---

## 五、命名规范

```
{级别}_{序号}_{格式类型}.json

级别：s1（S1样本）或 s2（S2样本）
序号：两位数字，从01开始
格式：weekly / monthly / ledger

示例：
  s1_01_weekly.json    # S1样本1，周报
  s2_01_monthly.json   # S2样本1，月报
  s2_02_ledger.json    # S2样本2，台账
  s2_03_weekly.json    # S2样本3，周报
```

---

## 六、提交方式和存档位置

**提交给：** PMO协调员（PC）

**PC收到后存档到：**
```
project/data/samples/s1/   # S1样本
project/data/samples/s2/   # S2样本
```

**PC同时更新registry.json：**
```json
{
  "last_updated": "ISO8601",
  "samples": [
    {
      "id": "s1_01_weekly",
      "level": "S1",
      "file": "s1/s1_01_weekly.json",
      "result": "待验收",
      "date_added": "日期",
      "added_by": "责任人姓名"
    }
  ]
}
```

---

## 七、PC验收标准（5维）

| 维度 | 通过标准 |
|------|---------|
| 字段完整性 | 全部必填字段非空 |
| 时间信息 | promised_items含时间词 |
| 状态可判定 | current_progress含判断关键词 |
| 责任人 | owner字段有内容 |
| 文字可解析 | 文字>10字符，非纯符号 |

**验收结果：**
- ✅ 通过 → 进入样本库
- ⚠️ 部分通过 → 降级使用，注明缺口
- ❌ 不通过 → 退回责任人补充

---

## 八、样本不足时如何处理

| 缺口 | 处理方式 |
|------|---------|
| 只有S0，无S1 | 暂用S0进行冒烟测试开发，样本到位后替换 |
| S1不足（<1份） | 阶段1关闭阻断，等待样本到位 |
| S2不足（<3份） | 阶段4进入阻断，等待样本到位 |
| 样本质量差 | 退回补充，不降级接受 |

---

## 九、责任人和PC的最短行动清单

**责任人（FU）：**
1. [ ] 按格式准备样本JSON
2. [ ] 执行脱敏（不得脱敏判断关键词）
3. [ ] 提交给PC

**PMO协调员（PC）：**
1. [ ] 收到样本后执行5维验收
2. [ ] 通过后存入对应目录
3. [ ] 更新registry.json
4. [ ] 通知Agent样本已到位
