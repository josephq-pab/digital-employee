## 演示脚本（按顺序执行）

### 开场（30秒）

**目标：建立背景，让听众知道我们在做什么**

```
"我们今天展示的是"组织执行力数字员工"的最小可运行原型。
输入：某总行PMO的周报（3期）+ 项目台账
输出：领导摘要 + 事项总台账 + 风险清单
整个过程约3秒完成。
这里有一个关键设计原则：我们不追求全自动，
所有判断都附了规则依据，人工可以复核、可以推翻。"
```

---

### 第一步：看输入数据（1分钟）

**目标：展示真实数据，增强可信度**

```
"让我们先看输入。这是2025年12月的三周周报，
每期包含key_alerts（黄灯事项）和closure_requests（结项申请）。
同时也看到，台账中记录了每个项目的进展和负责人。
我们在做的事，就是把这三份周报和台账打通。"
```

**展示内容：**
- 打开 `samples/samples/weekly_reports/weekly_report_series_latest3.json`（可选）
- 或者直接展示事项总台账中的"来源"列

---

### 第二步：看归一化结果（1分钟）

**目标：展示AI如何将不同格式的数据标准化**

```
"周报是周报格式，台账是台账格式，
数字员工第一步是把它们统一成标准事项格式。
11个事项被识别出来，包括4个黄灯和7个结项申请。
每个事项都有统一的字段：事项名称/期间/来源/责任人/承诺/进展/风险点。
（展示归一化JSON的前几行）
这就是数字员工处理不同格式的思路：适配层先行，统一格式再判断。"
```

**展示内容：**
- `normalized_20260407_220628.json`（部分）

---

### 第三步：看领导摘要（2分钟）

**目标：展示核心输出，展示三层分离**

```
"这是领导摘要，面向管理层。
我们来看黄灯事项——汽车行业和学校行业被识别出来，
判断依据是"责任人自述风险；进度滞后"。
状态是"关注"，风险等级是"中"。
重要：判断依据里既有触发规则名称（rule_进度滞后），
也有原始信号（"进度滞后"这个词来自原始文本）。
这意味着，管理者可以追问：这个判断是怎么来的？
——答案是"因为进度滞后这个词出现在了原始进展描述里"。
这就是"可复核"的含义。"
```

**展示内容：**
- `领导摘要_20260407_220628.txt`（黄灯事项部分）

---

### 第四步：看事项台账（1分钟）

**目标：展示完整的可追溯性**

```
"事项台账是完整的内部清单。
每一行对应一个事项，包含了：来源（周报还是台账）、
映射情况（有没有对应到台账记录）、
判断规则、原始信号。
比如：制定贷款促投放提升方案，对应到台账ledger-31，
判断结果：风险低，正常申请结项。
而汽车行业这一项，在台账里没有精确映射，
系统标注了"未映射"，后续需要人工确认。"
```

**展示内容：**
- `事项总台账_20260407_220628.txt`（选择几条典型）

---

### 第五步：看风险清单（1分钟）

**目标：聚焦重点，建立信任**

```
"风险清单是4项，全部是黄灯关注。
这里要特别说明：7个结项申请，已经正确地不进入风险清单，
因为它们代表"风险已释放"，不是"风险存在"。
这体现了数字员工的判断逻辑——
不是简单地扫描关键词，而是理解事项的语义类别。"
```

**展示内容：**
- `风险清单_20260407_220628.txt`（4项）

---

### 第六步：总结和边界（1分钟）

**目标：诚实说明限制，建立信任**

```
"总结一下：
✅ 最小闭环已打通——周报→归一化→判断→摘要→台账→清单
✅ 判断可追溯——每个结论都有规则名称和原始信号
✅ 人工可复核——所有结论都可以被推翻
⚠️ 目前仍是"条件性实现"——PMO口径和管理层格式还在确认中
⚠️ 行业→项目映射还有缺口——约67%的事项没有精确映射

下一步：修复完这些问题，就可以进入MVP试点。
谢谢大家。"
```
