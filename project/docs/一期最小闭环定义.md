# 一期最小闭环定义

## 完整链路描述

```
【输入】            【处理】              【人工】        【输出】         【使用场景】
                   ┌──────────────────────┐
PMO填报            │                      │
周报/台账          │  1. 字段解析          │                           管理层
（含项目名称、      │  2. 事实提取          │                           晨会/周报
 责任人、承诺事项、→ 3. 规则推理           │→ 人工审阅  → 结构化摘要  → 审阅
 当前进展、风险点）│  4. 风险判定          │  （必要环节）   （文本）  风险清单
                   │  5. 建议生成          │                           存档备查
                   │  6. 依据标注          │
                   └──────────────────────┘
                            ↑
                     纯规则推理
                     无LLM依赖
                     无人工干预
                     处理过程
```

## 逐环节说明

### 环节1：输入

**触发方式：** PMO 或项目责任人提交周报/月报（格式为结构化 JSON 或等效表单）

**最小字段集（必须）：**
- 项目名称
- 责任人
- 承诺事项（本期承诺完成的工作）
- 当前进展（本期实际完成情况）
- 风险点（自述，非必须）
- 是否请求领导介入（布尔值，默认否）

**数据来源：** 本期仅支持人工填报，不做自动采集

---

### 环节2：处理（系统自动，无需人工）

**处理内容：**
1. **字段解析**：读取输入各字段，空字段按默认值处理
2. **事实提取**：将输入中的关键描述原样透传为"事实清单"（不做推断）
3. **规则推理**：按《一期术语与判断口径》执行规则匹配
4. **风险判定**：基于规则结果给出风险等级（低/中/高）和升级建议
5. **建议生成**：基于风险等级和判定依据，生成建议动作列表
6. **依据标注**：在输出中明确记录触发本次判断的原始关键词

**系统不做的动作：**
- 不修正输入数据
- 不补充输入中缺失的语义
- 不对风险自述做真伪判断
- 不替代人工做升级决定

---

### 环节3：人工审阅（必要环节，不可跳过）

**审阅内容：**
- 确认系统提取的事实是否准确
- 确认判断依据是否合理
- 确认升级建议是否采纳
- 如需调整，直接在摘要上手动标注

**审阅角色：** 管理层或PMO指定负责人

**审阅时机：** 系统输出后，汇总报告使用前

---

### 环节4：输出

**输出格式：** 结构化 JSON + 可读文本摘要

**输出包含内容：**
- 项目名称（透传）
- 状态摘要（总体等级 + 进展摘要）
- 风险评估（等级 + 触发条件 + 摘要）
- 建议动作（优先级列表）
- 升级建议（是否建议，附依据）
- 判断依据（供人工复核的事实清单 + 规则名称 + 原始关键词）

---

### 环节5：使用场景

| 场景 | 输出形式 | 使用方 |
|------|---------|--------|
| 管理层晨会/周报审阅 | 打印/投影摘要文本 | 管理层 |
| PMO存档备查 | 结构化JSON存档 | PMO |
| 风险清单同步 | 摘要文本 + 风险列表 | PMO |
| 下一周期跟踪 | 与下期输出对比 | PMO/管理层 |

---

## 闭环完整性声明

**本闭环完整性条件：**
1. 输入字段完整 → 2. 系统处理完成 → 3. 人工审阅 → 4. 输出交付使用

**任一环节缺失，闭环不完整：**
- 无输入 → 无法处理
- 无人工审阅 → 不得将系统输出作为正式依据
- 无输出使用 → 闭环无实际业务价值

**一期结束后，闭环质量以"管理层实际使用并反馈"为最终验证标准。**
