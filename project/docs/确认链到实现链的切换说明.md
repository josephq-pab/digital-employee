# 确认链到实现链的切换说明

**版本：** v1.0
**日期：** 2026-04-07
**用途：** 说明从4A.7切到4B时，哪些文档视为冻结版；如何避免确认完成后又重新讨论边界

---

## 一、切换的核心原则

**一旦确认完成，边界即冻结。确认链闭合 = 实现链开始，不得再重新讨论边界。**

---

## 二、切换触发条件

切换触发的唯一条件：
- G1 = 🟢（PMO-01 + PMO-02 + ADPT-02 + ADPT-03 全部确认）
- G2 = 🟢（MGT-01 确认）
- G3 = 🟢（PILOT-01 确认）
- G4 = 🟢（Q01确认后自动升级）
- G5 = 🟢（Q01确认后自动升级）
- 总体判定 = 🟢

**任意一项不满足 = 不得切换，继续确认链阶段**

## 四、切换时的"最后讨论窗口"

**切换前设置7个工作日的"最后讨论窗口"：**

```
确认完成日期：__________
讨论截止日期：确认完成日期 + 7个工作日

截止日前：
  → 所有对边界/规则/格式的异议必须提出
  → 逾期视为无异议
  
截止日后：
  → 不得再提出新的边界讨论
  → 如有重大问题，走正式变更申请流程
```

---

## 五、变更申请流程（切换后仍可申请，但需审批）

### 5.1 可接受的变更申请

| 变更类型 | 示例 | 审批人 |
|---------|------|-------|
| 轻微措辞调整 | 文字描述更清晰，不改实质 | Agent自行判断 |
| 新增字段（不影响规则） | 增加备注字段 | PMO确认 |
| 展示格式微调 | 调整输出字段顺序 | PMO确认 |
| 冒烟测试新增用例 | TC21-TC25 | PMO确认 |

### 5.2 不可接受的变更申请（禁止）

| 变更类型 | 示例 | 理由 |
|---------|------|------|
| 核心规则修改 | 修改严重滞后的判断条件 | 违反冻结原则 |
| 删除已有字段 | 删除risk_points字段 | 影响判断逻辑 |
| 修改输出格式结构 | 取消三层结构 | 影响管理层已确认的格式 |
| 推翻已确认的术语定义 | 将"滞后"定义为其他含义 | 违反Q01确认结果 |

## 八、切换时的注意事项

1. **不要在切换时重新讨论已确认项**：切换点是终点，不是新的讨论起点
2. **不要因为"感觉不对"就暂停切换**：已确认=已确认，需走变更流程
3. **不要跳过切换直接开始开发**：门控未清零=禁止开始4B
4. **切换后仍保持门控更新机制**：G6/G7的变化仍需更新门控卡
