# 正式MVP试跑首周SOP

**版本：** v1.0
**日期：** 2026-04-08
**适用范围：** 正式MVP试跑启动后首周（Day 1 - Day 7）

---

## 一、总体节奏

| 阶段 | 时间 | 核心任务 |
|------|------|---------|
| 启动 | Day 1 | 收样 → 运行 → 复核 → 问题登记 → 发出首份周报 |
| 稳定运行 | Day 2-5 | 持续收样 → 执行 → 观察输出 → 收集反馈 |
| 周结论 | Day 7 | 汇总本周数据 → 判断是否继续 → 发出首周结论 |

---

## 二、每日操作流程

### Day 1：正式启动

**上午**
1. 确认样本到位（周报+台账+映射）
2. 确认角色分工已就位
3. 确认门控全🟢

**下午**
4. 执行试跑：`python project/scripts/run_prototype.py`
5. 检查三件套是否全部生成
6. 复核负责人对照原始样本检查输出
7. 登记问题（如有）
8. 发出第一份《正式MVP试跑日报》

### Day 2-3：稳定运行

**每日**
1. 收集新样本（如果有新周报/台账）
2. 执行试跑
3. 快速复核（重点：黄灯/风险项是否正确）
4. 如有新问题，登记到《MVP试跑问题登记表》
5. 记录当日运行情况

### Day 4-5：持续观察

1. 持续执行
2. 收集复核负责人反馈
3. 汇总本周问题
4. 初步判断是否需要修复

### Day 6：周报准备

1. 汇总本周执行情况
2. 整理问题清单
3. 起草周报

### Day 7：周结论

1. 发出《正式MVP试跑周报》
2. 判断是否建议继续/暂停/调整
3. 如需调整，明确修复计划

---

## 三、每个操作节点详解

### 3.1 收样

| 项目 | 说明 |
|------|------|
| 样本来源 | PMO提供（人工提交，不自动采集） |
| 样本格式 | JSON（周报/台账/映射） |
| 放置路径 | samples/samples/weekly_reports/、samples/samples/project_ledger/、samples/samples/linked/ |
| 检查 | 样本非空，JSON可解析 |

### 3.2 运行

| 项目 | 说明 |
|------|------|
| 命令 | `python project/scripts/run_prototype.py` |
| 期望输出 | 领导摘要 + 事项台账 + 风险清单 + normalized.json |
| 异常判断 | 脚本报错 / 冒烟测试<5/7 / 三件套缺失 |

### 3.3 人工复核

| 项目 | 说明 |
|------|------|
| 复核重点 | 风险信号是否合理？结项判断是否正确？owner是否匹配？ |
| 异常判断 | 有明显误判 / 重大遗漏 / 格式混乱 |
| 复核时间 | 建议10-15分钟 |

### 3.4 问题登记

| 项目 | 说明 |
|------|------|
| 登记位置 | 《MVP试跑问题登记表》 |
| 分类 | 阻断型/继续型/展示型/数据型/规则型 |
| 截止时间 | 当日17:00前 |

### 3.5 周结论

| 项目 | 说明 |
|------|------|
| 周报提交时间 | 每周最后一个工作日 |
| 周报内容 | 本周输入/输出/问题/判断 |
| 最终判断 | 继续/暂停/调整 |

---

## 四、角色分工（首周）

| 角色 | 核心任务 |
|------|---------|
| 样本负责人 | 确保样本每日到位 |
| 运行负责人 | 执行脚本、记录日志、收集输出 |
| 复核负责人 | 人工复核、问题登记 |
| 问题登记人 | 汇总问题、维护登记表 |
| 阶段判断人 | Day 7给出周结论 |

---

## 五、升级机制

如在运行过程中发现以下情况，立即触发升级：

| 情况 | 动作 |
|------|------|
| 脚本报错无法运行 | 运行负责人立即通知阶段判断人 |
| 发现P0级问题 | 暂停试跑，触发回退机制 |
| 输出结果与样本严重不符 | 复核负责人立即通知阶段判断人 |
