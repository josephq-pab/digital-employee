# 一期原型技术方案

**版本：** v1.0
**目的：** 指导阶段4原型实现的工程蓝图，不写代码，但达到可指导开发的细度

## 一、整体架构：五层结构

```
┌─────────────────────────────────────────────────────┐
│  第5层：使用层                                        │
│  管理层晨会审阅 / PMO存档 / 风险清单同步               │
├─────────────────────────────────────────────────────┤
│  第4层：输出层                                        │
│  结构化JSON + 可读文本摘要（领导摘要/事项台账/风险清单） │
├─────────────────────────────────────────────────────┤
│  第3层：判断层（核心）                                │
│  规则引擎 → 风险判定 → 建议生成 → 依据标注            │
├─────────────────────────────────────────────────────┤
│  第2层：解析层                                        │
│  字段解析 → 事实提取 → 输入校验                       │
├─────────────────────────────────────────────────────┤
│  第1层：输入层                                        │
│  支持：JSON结构化输入（周报/月报/项目台账格式）        │
└─────────────────────────────────────────────────────┘
         ↑
   人工填报（不可跳过）
   人工审阅（不可跳过）
```

## 二、输入层（第1层）

### 2.1 支持的输入格式

**格式A：周报/月报格式**

```json
{
  "project_name": "智慧园区一期",
  "owner": "张明",
  "report_type": "weekly",       // weekly | monthly
  "report_date": "2026-04-07",
  "promised_items": "本周承诺：完成权限模块开发",
  "current_progress": "权限模块开发完成，数据看板开发滞后约2周",
  "risk_points": "数据源接口对接进度不及预期",
  "escalation_requested": false
}
```

**格式B：项目台账格式**

```json
{
  "project_name": "智慧园区一期",
  "owner": "张明",
  "last_updated": "2026-04-07",
  "items": [
    {
      "item_id": "M1",
      "promised": "6月30日前完成基础平台部署",
      "current_progress": "完成80%",
      "status": "in_progress"
    }
  ],
  "overall_risk_points": "数据源接口对接进度不及预期",
  "escalation_requested": false
}
```

### 2.2 输入容错规则

| 场景 | 处理方式 |
|------|---------|
| risk_points为空 | 按"无风险自述"处理 |
| escalation_requested缺失 | 默认为false |
| report_type缺失 | 默认为weekly |
| items字段缺失 | 降级为格式A处理（promised_items字段） |
| 所有字段完整但某字段过长 | 按字段长度上限截断（保留前2000字符） |

## 三、解析层（第2层）

### 3.1 字段解析

**任务：** 将输入映射为标准内部表示（InternalProjectRecord）

| 内部字段 | 来源（格式A） | 来源（格式B） |
|---------|------------|-------------|
| project_name | project_name | project_name |
| owner | owner | owner |
| promised_items | promised_items | 合并items[].promised |
| current_progress | current_progress | 合并items[].current_progress |
| risk_points | risk_points | overall_risk_points |
| escalation_requested | escalation_requested | escalation_requested |

### 3.2 事实提取

**任务：** 将输入中的描述性文本，原样透传为事实清单，不做推断。

提取规则：
- promised_items → "承诺事项：{原文}"
- current_progress → "当前进展：{原文}"
- risk_points非空 → "风险自述：{原文}"
- escalation_requested=true → "责任人主动请求领导介入"

### 3.3 输入校验

| 检查项 | 通过条件 | 不通过时 |
|--------|---------|---------|
| project_name非空 | 字符串长度>0 | 报错，不处理 |
| owner非空 | 字符串长度>0 | 报错，不处理 |
| promised_items非空 | 字符串长度>0 | 报错，不处理 |
| current_progress非空 | 字符串长度>0 | 标记为"无明确进展" |

## 四、判断层（第3层，核心）

### 4.1 规则引擎设计

**引擎输入：** InternalProjectRecord（标准化内部记录）
**引擎输出：** JudgementResult（判断结果）

**规则匹配顺序（重要）：**

```
1. 先检查 escalation_requested == true → 触发"高风险/建议升级"，跳过后续
2. 再检查 progress 严重滞后 → 触发"高风险/建议升级"
3. 再检查 progress 滞后 + risk_points 非空 → "中风险/不升级"
4. 再检查 progress 滞后 → "中风险/不升级"
5. 再检查 risk_points 非空含关键词 → "低风险/不升级"
6. 以上均不触发 → "低风险/不升级"
```

### 4.2 状态机

```
输入状态      →  规则判定  →  风险等级  →  升级建议
(null/空)      →  默认规则  →  低        →  不升级
有progress    →  匹配滞后   →  中/高     →  视规则而定
有risk_points →  匹配风险词  →  低/中     →  视规则而定
escalation=1  →  强制升级   →  高        →  建议升级
```

### 4.3 建议生成

**规则：**
- 风险高 + escalation=true → "建议尽快向领导汇报，获取资源支持"
- 风险中 → "分析滞后根因，制定缓释措施"
- 风险低 → "继续保持，定期更新"
- 有风险自述 → "对风险自述进行深入分析，制定缓释措施"

## 五、输出层（第4层）

### 5.1 三种输出形态

| 输出 | 内容 | 格式 | 使用场景 |
|------|------|------|---------|
| 领导摘要 | 短文本（≤500字） | 纯文本 | 晨会投影/打印 |
| 事项总台账 | 结构化事项列表 | JSON数组 | PMO存档/跨周期对比 |
| 风险清单 | 风险项列表 | JSON数组 | 风险专项汇报 |

**一期原型要求：** 至少实现领导摘要输出。事项总台账和风险清单作为扩展输出在OUTPUT_SCHEMA中定义，阶段4可不实现。

### 5.2 领导摘要模板

```
项目：{project_name}
责任人：{owner}
总体状态：{overall}（{risk_level}风险）
进展摘要：{progress_text}
风险提示：{risk_summary}（触发条件：{triggered_by}）
建议动作：{suggested_actions.join("；")}
升级建议：{escalation_recommended ? "建议升级关注" : "暂不需要升级"}
判断依据：{reasoning.raw_signal}
```

## 六、使用层（第5层）

### 6.1 人工审阅节点（不可跳过）

**位置：** 系统输出之后，正式使用之前

**审阅内容：**
- 事实提取是否准确
- 风险等级是否合理
- 升级建议是否采纳
- 有无需要手动调整的内容

**审阅后：** 可直接使用，或手动标注后使用

## 八、技术选型约束

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 推理方式 | 纯规则（非LLM） | 一期稳定可复核 |
| 实现语言 | Python 3 | 快速原型，零依赖 |
| 输入格式 | JSON | 结构化，无歧义 |
| 无外部数据库 | ✅ | 一期不做持久化 |
| 无前端界面 | ✅ | 一期只输出文本/JSON |
