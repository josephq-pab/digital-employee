## 一、整体定位

对内演示的核心是建立信任——让技术人员和PMO理解数字员工的判断逻辑、规则机制和人工确认节点，从而愿意使用它、帮助完善它。

## 二、讲解词

### 开场：我们在做什么

```
"数字员工是执行层面的AI助手，它不是来取代PMO的，
而是来替代重复劳动的。
具体来说：给定一份周报，它能自动识别风险事项、
判断风险等级、生成摘要，让管理层开会有东西看。

不追求全自动——每一项判断都有规则依据，
人工可以复核、可以推翻。这是我们这期设计的核心原则。"
```

### 技术实现路径

```
"一期我们选择了纯规则引擎，不是LLM。
原因很简单：规则引擎的判断是可测试的、可复核的。
TC01-TC05五类场景，冒烟测试全部通过。
加规则、加case，不会破坏已有判断。
这是我们认为当前阶段最稳妥的路径。"

（可展示：project/app/engine.py 的LAG_PATTERNS）
```

### 三层分离机制

```
"数字员工的输出严格分三层：

第一层【事实】：原始数据，承诺什么、进展到什么程度
第二层【判断】：规则名称 + 原始信号，告诉你是哪条规则得出了这个结论
第三层【建议】：建议动作，是系统给出的行动推荐，不是最终决策

举例：黄灯汽车行业，判断是"中风险"，
依据是：rule_进度滞后 + rule_risk_self_reported，
原始信号是：进度滞后（这4个字出现在current_progress字段）
→ 人工看到后，可以同意，也可以说"这个判断不符合我的判断"，
  然后追溯是哪条规则出了问题，这是我们说的"可复核"。"
```

### 适配层机制

```
"周报是一种格式，台账是另一种格式，
数字员工通过"适配层"把它们统一成标准格式再处理。
适配层处理的事情包括：
- 周报没有owner字段，用台账的owner_department替代（provisional）
- escalation_requested字段周报没有，用management_summary推断（provisional）
- 黄灯信号默认注入"进度滞后"到current_progress（QUAL-01修复）

适配层不修改规则引擎，只负责格式转换和信号增强。
这是我们实现"最小闭环"的核心架构。"
```

### 人工确认节点

```
"我们在两个地方保留了人工确认节点：

节点1：判断输出之后——管理者可以推翻AI判断
节点2：映射关系——行业和项目的对应需要PMO确认

这不是技术限制，是一期设计原则：
"允许人工确认，不追求全自动"。
目的是让AI和人协同，而不是让人被AI绑架。"
```

### 限制与边界

```
"目前阶段已知限制：

1. 周报本身没有owner字段，我们用台账的owner_department替代
   → 约67%的映射缺口，根源在此
2. 行业维度和项目维度不同构
   → "汽车行业"是周报的行业视角，
   → "六策合一-汽车零部件"是台账的项目视角，
   → 需要PMO确认映射关系
3. 目前所有规则都是"条件性实现"，正式版本等PMO确认
4. 目前不接生产系统，不做平台集成

这些限制我们不藏着掖着，都是明面上的。"
```

### 后续路径

```
"下一步分两步：

第一步（近期）：收到PMO对术语口径的确认后，
把"条件性实现"升级为"正式版本"，然后进入试点。

第二步（试点期）：在1-2个真实项目上跑，
看判断准确率，收集反馈，继续迭代。

我们的目标不是做一个完美系统再上线，
而是做出最小可用原型，在试点中持续完善。"
```

## 三、常见问题应对

| 问题 | 应对 |
|------|------|
| 为什么不用LLM？ | 规则引擎可测试、可复核；LLM在判断一致性上有不确定性 |
| 映射缺口怎么解决？ | 需要PMO确认MAP-01~04，行业和项目的对应关系是业务判断 |
| 判断能被推翻吗？ | 能。数字员工输出后，管理者可以修改，这是设计的一部分 |
| 什么时候接生产系统？ | 一期目标不接生产，MVP验收后才考虑 |
